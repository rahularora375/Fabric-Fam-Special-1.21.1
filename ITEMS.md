# ITEMS.md

Per-theme item details, signature-weapon convention, and "Adding new items" patterns. Root context: `CLAUDE.md`. For non-item systems referenced here (`ArmorEffects`, `AttackHandlers`, mixins, etc.), see `SYSTEMS.md` and `MIXINS.md`.

## Theme overview

Items are grouped by theme under `item/entries/`. Items with passive effects set their component flags on the stack in these files before adding to the creative tab, e.g. `stack.set(ModComponents.GRANTS_NIGHT_VISION, true)`.

**Universal identity flag.** Every themed stack (weapons, tools, armor — all 11 themes including OldFam) carries `IS_FAMSPECIAL_GEAR=true`. Themed sets (those carrying `SET_ID`) total 10 (OldFam excluded)., stamped automatically by `ModItemGroups.buildItem` at creation time. This is the forgery-protection anchor for the chestplate→elytra merge (see [Chestplate→Elytra Merge Feature](#chestplateelytra-merge-feature)) — the mixin rejects any left-slot chestplate without it, so players can't smuggle a vanilla diamond chestplate with a manually-stamped `SET_ID` through the anvil.

- **`OldFamItems`** — 13 legendary diamond gear pieces (weapons, tools, armor with fixed-level enchants).
- **`MistbornItems`** — Atium Dagger (netherite sword with day/night attribute swap) + 4-piece Mistborn diamond armor set. Each piece carries a flat +1 MAX_HEALTH (0.5 heart) `AttributeModifiersComponent` entry on top of the vanilla diamond baseline — 2 hearts total across the set. Strength I + Speed I fire at night or in Nether/End as a hidden, time-gated payoff.
- **`PacifistItems`** — pink + white pacifist theme. Sage's Grace diamond sword (Unbreaking 5 + Mending, ships with a fixed loadout) that heals its target on hit; subject to vanilla anvil / grindstone / smithing-table rules in both creative and survival (the older mixin-layer modification lockouts have been removed). Plus a 4-piece diamond armor set — Sage's Crown / Viking's Farewell / Sage's Wrap / Son of Thors. Each piece carries a flat +2 MAX_HEALTH (1 heart) — 4 hearts across the set. The set also grants Regeneration I while 4/4 worn; Sage's Crown additionally flags `SHOWS_ENTITY_HP` so the wearer sees floating HP bars above other players. No time gate.
- **`PoseidonItems`** — deep-sky-blue + gold "god of the sea" theme. Trident of Olympus (Unbreaking 4, Riptide 5, Impaling 5 — fixed-loadout, no Mending) + 4-piece diamond armor set (Lorelai's Crown / Ironborn Plate / Highstorm's Bind / Phinn's Trudge, all tagged `SET_ID="poseidon"`). - **`FireSerpentItems`** — neon-green + blazing-orange "serpent's wrath" theme. Fire Serpent's Wrath diamond spear (Lunge 3, Fire Aspect 2, Mending — no Sharpness/Looting/Unbreaking; flagged `NO_LUNGE_HUNGER` so lunging doesn't eat hunger) + 4-piece diamond armor set (Impaler's Crown / Serpent's Embrace / Kindling's Wrap / Kindling's Path, all tagged `SET_ID="fire_serpent"`). Each piece carries a flat +1 MAX_HEALTH (0.5 heart) and +0.0025 MOVEMENT_SPEED — 2 hearts + 10% move speed at 4/4. Impaler's Crown additionally flags `GRANTS_MESSMERS_FLAME` (Messmer's Flame Aegis — 80% fire damage reduction via `LivingEntityMixin`). The 4/4 set bonus is **Messmer's Venom** — the wearer's spear hits apply a custom poison effect that bypasses vanilla poison immunities (works on undead, spiders, the Wither, the Ender Dragon, etc.).
- **`NecromancerItems`** — muted-green + crimson "Michael-Jackson-meets-necromancer" theme (MJ-song-reference naming). Thriller's Edge netherite axe (Sharpness 5, no other enchants, flagged `APPLIES_WITHER_ON_HIT` for Wither II on melee hit; +0.2 `ATTACK_SPEED` on top of the vanilla netherite-axe baseline of +9 ATTACK_DAMAGE / -3.0 ATTACK_SPEED, netting the player to 1.2 attack speed) + 4-piece diamond armor set (Fedora of the Damned / Smooth Criminal's Vestment / Billie Jeans / Moonwalker's Curse, all tagged `SET_ID="necromancer"`). Lives in `NecromancerSummon` (event-driven; no `BONUSES` entry). The `ROTTEN_MUSCLE` HUD badge (display name "Rotten Muscle") advertises the summon-charged state and disappears during the cooldown window; each worn piece's tooltip also carries a crimson `MM:SS` countdown read off the `NECROMANCER_COOLDOWN_END` component.

Just Hit Bro diamond pickaxe (Fortune 4, no set membership, flagged `BONUS_DIAMOND_CHANCE`) + 4-piece diamond armor set (Time Dekhlo Helmet / Bombardiro Coccodrillo / Knee-Use Paper / Dhoom Machale, all tagged `SET_ID="esh_endra_navesh"`). Just Hit Bro carries `BONUS_DIAMOND_CHANCE`: every successful block break rolls a 1/800 chance to additionally phantom-drop vanilla's `blocks/diamond_ore` loot (Fortune + Silk Touch apply naturally since the same pickaxe stack is passed in) and play `ModSounds.BONUS_DIAMOND`. Advertised via the `SHADI_BUFF` HUD badge (gold `0xE8D171`) while the pickaxe is in the main hand.
- **`ShurimaItems`** — worn-gold + pale-sand "Sun Emperor" theme (palette: prefix `#C8A84B` italic, piece `#F0E6C8` bold, lore base `#B8A070` italic, accent `#E8922A` warm orange). Sun Disc Spear (theme-adjacent, no `SET_ID`; built on `Items.DIAMOND_SPEAR`, fixed Knockback 5 + Sharpness 6, flagged `IGNORES_KB_RESISTANCE`) + 4-piece diamond armor set (Sun Emperor's Crown / Mantle of Shurima / Sandstrider Wraps / Drifting Sands, all tagged `SET_ID="shurima"`). Each piece carries a flat +1 MAX_HEALTH (0.5 heart), +0.0025 MOVEMENT_SPEED, **and +0.075 KNOCKBACK_RESISTANCE** on the piece's slot — totals at 4/4 are +2 hearts, +0.01 move speed (10% of base), +0.3 KB resistance (30% stiffness). Drifting Sands (the boots) additionally flags `GRANTS_SUNS_PROTECTION` (boots-alone bonus, gated on **desert-family biome** — `BiomeKeys.DESERT`, `BADLANDS`, or `ERODED_BADLANDS` — applies the `SUNS_PROTECTION` effect via `ArmorEffects` for 20% incoming-damage reduction in `LivingEntityMixin`). The 4/4 set bonus is **Shuriman Endurance** — desert-family-gated, 4-minute apply-once `SHURIMAN_ENDURANCE` effect that cancels every `PlayerEntity#addExhaustion` call at HEAD via `PlayerEntityExhaustionMixin` (no saturation loss anywhere from any source). Beacon-style: once earned in any desert-family biome, the buff persists wherever the player walks for the full ~4 minutes — apply-once gating in `ArmorEffects` skips re-application while remaining duration > 4720 t so leaving the biome doesn't reset the timer. Sun Disc Spear's `IGNORES_KB_RESISTANCE` drives **Emperor's Divide** (per-tick fast-path bonus → `EMPERORS_DIVIDE` effect → `LivingEntityKnockbackMixin` zeroes the target's `KNOCKBACK_RESISTANCE` lookup so a hit always lands full-strength knockback regardless of target stiffness).

Each piece carries a flat +1 MAX_HEALTH (0.5 heart) — 2 hearts at 4/4. Trousers of the Trail additionally flag `BOUNTY_HUNTER` (drives the leggings-gated arrow-kill diamond drop in `BountyHunterKills`). The 4/4 set bonus is **Technoblade Never Dies** — fall-damage immunity, applied via the `TECHNOBLADE_NEVER_DIES` effect and read by a 5th `@ModifyReturnValue` in `LivingEntityMixin` that returns 0.0f when the source is `DamageTypes.FALL`. The crossbow's **Crusader's Volley** mainhand bonus fires **three shots** from a single right-click charge at a snappy ~100 ms gap between each (2-tick cadence — shot 1 at t=0, shot 2 at t=2, shot 3 at t=4, full burst spread ~200 ms), with all three shots matching the originally-loaded projectile type (arrows OR firework rockets — the loaded projectile is captured before `super.use(...)` clears the `CHARGED_PROJECTILES` component and stashed on the `ScheduledShot` so shots 2/3 can replay the right entity type). Iframe bypass: arrow-path shots 2 and 3 spawn a custom `VolleyArrowEntity` (in `entity/VolleyArrowEntity.java`) that zeroes the target's `LivingEntity.timeUntilRegen` before delegating to vanilla `onEntityHit`, so all 3 shots reliably damage the same target despite the cadence falling inside vanilla's 10-tick iframe window; firework-path shots 2 and 3 spawn a vanilla `FireworkRocketEntity` and rely on vanilla iframe rules. Shots 2 and 3 inherit Multishot/Piercing levels from the source stack — so Multishot III + Crusader's Volley = 9 arrows per release (defensive code; vanilla means of applying Multishot to Fortune & Glory are blocked separately). No extra ammo or durability cost beyond what vanilla took for shot 1.
- **`ThorItems`** — Asgardian gold + storm-blue theme. **5-piece set** (the only 5/5 set in the mod): Thunderhelm + 3 armor pieces + Mjolnir, all tagged `SET_ID="thor"`. Each of the 4 armor pieces (Thunderhelm, Asgardian Plate, Warrior's Greaves, Storm Strider) carries a flat **+1 MAX_HEALTH (0.5 heart)** — **+2 hearts total when the 4-armor set is worn**. Mjolnir does **not** contribute to the health bonus. The health bonus carries over onto the merged elytra via the attribute-transfer path (see [Chestplate→Elytra Merge Feature](#chestplateelytra-merge-feature)). Mjolnir is a `MjolnirMaceItem` (extends vanilla `MaceItem`) flagged `THOR_MACE` + `LIGHTNING_ON_HIT` and counts as the **5th** piece toward the set gate. Storm Strider boots ship with **Feather Falling IV** baked in via `buildItem`'s enchant varargs. Warrior's Greaves (the leggings) flags `TRIGGERS_STORM_AWAKENING` — drives **Storm's Awakening**: each kill while the leggings are worn rolls (currently `1.00f`, a TESTING VALUE still flagged as such in code pending ship) to start a thunderstorm, with a thunder sound (`SoundEvents.ENTITY_LIGHTNING_BOLT_THUNDER`, vol 10000, pitch 0.8) at the killer position immediately before `world.setWeather(...)`. Storm duration is **4800 t (~4 min real-time)** (was "one full Minecraft day"). Per-player **24000-tick (1 MC day)** cooldown only arms on a successful trigger; if `world.isThundering()` is already true, the kill is skipped entirely (no roll, no cooldown burn). On a successful trigger, `STORM_COOLDOWN_END` is stamped onto the worn Warrior's Greaves for the client tooltip's crimson "MM:SS" countdown — parallel to how `NECROMANCER_COOLDOWN_END` works for the Rotten Muscle line. Mjolnir's `LIGHTNING_ON_HIT` rolls per-hit lightning (100% during a thunderstorm, 20% in clear weather) via `ThorEffects.AFTER_DAMAGE`. Mjolnir ships with **Riptide III** at creation time (down from V). Mjolnir is **not** anvil-locked at the mixin layer — it behaves per vanilla mace anvil rules, in both creative and survival (the former `AnvilScreenHandlerMixin` TAIL guard that blanked Mjolnir+book outputs has been removed). **Mjolnir's enchant eligibility is "vanilla-mace-equivalent + Riptide only"**: accepts Density / Breach / Wind Burst / Smite / Bane of Arthropods / Fire Aspect / Mending / Unbreaking / Curse of Vanishing / Riptide, but **not** Sharpness (intentionally — `enchantable/sharp_weapon` is not extended) and **not** Loyalty / Channeling (still gated on the unmodified vanilla `enchantable/trident`). Driven by datapack tag overrides under `src/main/resources/data/minecraft/tags/item/enchantable/*.json` plus a vanilla-Riptide override pointing at the custom `famspecial:riptide_eligible` tag — see `SYSTEMS.md` "Datapack tag overrides" for the full mechanism. Each successful Mjolnir launch (charge + release, both dry-land Asgardian's Flight and vanilla water/rain riptide) costs **-1 durability** to Mjolnir, gated server-side. The **5/5 set bonus** is **Asgardian's Flight** — grants a dry-land riptide launch on Mjolnir right-click (charge ~10 ticks, then release; uses vanilla `TridentItem` Riptide math via `EnchantmentHelper.getTridentSpinAttackStrength(stack, player)` so launch magnitude scales with the Riptide level on the stack, now capped at III). Gated on `player.hasStatusEffect(ASGARDIANS_FLIGHT)` rather than re-deriving the set check, which itself is applied only while the four Thor armor pieces are worn AND Mjolnir is in the main hand — so breaking either half (armor swap or hotbar swap) instant-strips the effect via the `refreshBonusesFor` dispatch called from the equip / slot mixins and short-circuits the launch.

**Signature-weapon convention.** Each theme's signature weapon (Atium Dagger, Sage's Grace, Trident of Olympus, Fire Serpent's Wrath, Thriller's Edge, Oathbringer, Sun Disc Spear, Fortune & Glory) is theme-adjacent, not a set member — none carry `SET_ID`, so the 4/4 armor bonus only counts the four armor pieces. Weapons can still carry their own flags (`NIGHT_STRENGTH`, `HEALS_TARGET`, `NO_LUNGE_HUNGER`, `APPLIES_WITHER_ON_HIT`, `GRANTS_SHARDBEARING`, `CRUSADERS_VOLLEY`, etc.) independent of set membership, and the "Part of the X Set" tooltip line never appears on them. The same convention applies to Just Hit Bro (theme-adjacent pickaxe in the EshEndraNavesh theme — no `SET_ID`).

## Chestplate→Elytra Merge Feature

A cross-theme anvil recipe lets players fuse a themed diamond chestplate with a vanilla elytra, producing an elytra that retains the chestplate's identity — `SET_ID`, custom name, lore, all custom `ModComponents.*` flags, and the chestplate's custom attributes (MAX_HEALTH, MOVEMENT_SPEED, KNOCKBACK_RESISTANCE, etc.). The vanilla diamond `ARMOR` and `ARMOR_TOUGHNESS` attribute entries are **intentionally stripped** — the merged elytra trades armor points for flight. Implemented via a HEAD+cancellable `@Inject` on `AnvilScreenHandler.updateResult` in `AnvilScreenHandlerMixin.famspecial$mergeThemedChestplateElytra`; component transfer goes through `ModComponents.copyModComponentsForElytra(from, to)`.

**Eligible themes (10).** Mistborn, Pacifist, Poseidon, FireSerpent, Necromancer, KnightRadiant, EshEndraNavesh, Shurima, Thor, Raider. **OldFam is excluded** — OldFam pieces don't carry `SET_ID`, so they don't enter the merge path.

**Recipe.**
- Left slot: themed diamond chestplate. Must carry `IS_FAMSPECIAL_GEAR=true` (stamped automatically by `ModItemGroups.buildItem` at creation time). This is the forgery anchor — a vanilla diamond chestplate with a hand-stamped `SET_ID` NBT tag will not match.
- Right slot: vanilla elytra with no `SET_ID` (so already-merged elytras don't re-merge).
- Level cost: 4.
- Rename-field text is honored when non-blank; otherwise the chestplate's custom name is carried forward.

**Enchant filter.** Only **Mending** and **Unbreaking** survive on the output. Enchants from **both** inputs (source chestplate and vanilla elytra) are considered, with the **max level winning** when both contribute the same enchant — feeding a Mending chestplate + a Mending-II elytra yields Mending II on the output. All other enchants (Protection, Thorns, etc.) are stripped.

**Component transfer — `ModComponents.copyModComponentsForElytra`.** Carries every custom component in `MOD_COMPONENTS_BASE` **except `INDESTRUCTIBLE`** — redundant on an elytra, since vanilla already clamps elytra damage at `maxDamage - 1`. `REGENS_DURABILITY` **does** transfer (the merged elytra stays regen-managed). `BLOCKS_MENDING` also transfers, but is currently a no-op flag — after the recent anvil/grindstone refactor, nothing reads it; kept in the copy path for forward compatibility. Merged elytras are subject to vanilla anvil rules post-merge, so re-enchanting them with Mending via an enchanted book is possible if the player wishes.

**Attribute transfer.** The chestplate's `AttributeModifiersComponent` is filtered entry-by-entry: vanilla `ARMOR` and `ARMOR_TOUGHNESS` entries are dropped, and every other entry (custom MAX_HEALTH, MOVEMENT_SPEED, KNOCKBACK_RESISTANCE, etc.) copies over as-is. Per-theme effect: Mistborn = +0.5 hearts, Pacifist = +1 heart, Poseidon = +1 heart, FireSerpent = +0.5 hearts + 2.5% move speed, Necromancer = +0.5 hearts, EshEndraNavesh = +0.5 hearts, Shurima = +0.5 hearts + 2.5% move speed + 7.5% KB resist, Thor = +0.5 hearts, Raider = +0.5 hearts. KnightRadiant ships no attribute modifiers so its merged elytra gains no attributes, but still keeps `INDESTRUCTIBLE`/`REGENS_DURABILITY`/`BLOCKS_MENDING` via the component path (minus `INDESTRUCTIBLE`, which is filtered out).

**Set-bonus interaction.** The merged elytra still counts toward the 4-piece set-bonus checks driven by `ArmorEffects.hasFullSet(SET_ID)`, because those gates read `SET_ID` off the chest slot without an item-class check. Equipping the merged elytra plus the theme's helmet, leggings, and boots still lights up the 4/4 bonus; for Thor specifically, merged elytra + helmet + legs + boots + Mjolnir satisfies the 5/5 predicate.

**Adding a new `ComponentType`.** If a new custom component should be copied to the merged elytra, add it to `MOD_COMPONENTS_BASE` in `ModComponents`; `copyModComponentsForElytra` iterates that list minus the `INDESTRUCTIBLE` filter. Components not in that list silently won't transfer.

## Adding new items

Per user preference: **new items go only in `item/entries/*Items.java`** (the two creative tabs). Do not add them to `loot/Legendary*Pool.java` unless explicitly asked — loot-table additions are managed intentionally and separately.

**Identity flag.** All themed stacks automatically carry `IS_FAMSPECIAL_GEAR=true` via `ModItemGroups.buildItem`. Do not remove or override this — it is the forgery-protection anchor for the chestplate→elytra anvil merge and will likely gate future cross-theme features that need to distinguish real themed gear from vanilla items with spoofed NBT.

**Tab split.** Each theme file exposes three public methods — `addWeapons(entries, enchants)`, `addTools(entries, enchants)`, `addArmor(entries, enchants)` — with empty bodies where a theme doesn't ship that category (e.g. EshEndraNavesh has no weapons, Mistborn has no tools). The **Gear** tab's `.entries(...)` lambda in `ModItemGroups.GEAR_GROUP` calls every theme's `addWeapons` first (all weapons across themes, grouped by type), then every theme's `addTools`. The **Armor** tab's lambda in `ModItemGroups.ARMOR_GROUP` calls every theme's `addArmor` in theme order. Within a theme, follow vanilla page order: weapons = sword(s) → axe → bow → trident; tools = pickaxe → shovel → hoe; armor = helmet → chestplate → leggings → boots. When adding a new item, drop the `entries.add(...)` call into the method matching its type in the right position.

Common patterns:

- **Simple worn-effect flag** (Night Vision Helmet pattern): add a `ComponentType` in `ModComponents` → `stack.set(...)` it in the theme's `addArmor` → add a `Bonus.effects(...)` entry to `ArmorEffects.BONUSES` with a matching `TickContext` predicate → add a tooltip line in `FamSpecialClient`.
- **Set bonus**: set `SET_ID` to a shared string on each piece. For the **status-effect** half (Regen/Strength/Speed/etc.), add a `Bonus.effects(...)` entry to `ArmorEffects.BONUSES` whose trigger calls `ctx.hasFullSet("yourSetId")` (optionally combined with time/dimension gates). For the **attribute** half (+max health, armor, etc.), follow the Mistborn/Pacifist pattern: define a single flat `AttributeModifiersComponent` constant per slot in the theme file (start the builder from `new ItemStack(Items.DIAMOND_HELMET).get(DataComponentTypes.ATTRIBUTE_MODIFIERS).modifiers()` to preserve vanilla armor/toughness, then append your own entries with unique `Identifier`s and the correct `AttributeModifierSlot`). Apply the constant on each piece once in `addArmor`. No swap logic needed — values render natively. Add a matching branch in `FamSpecialClient`'s tooltip callback (reuse `buildSetLine(setName, count, active)` + `grantLine(...)` for status effects; attribute numbers render natively).
- **Main-hand attribute swap** (Atium Dagger pattern): define two `AttributeModifiersComponent` constants in the theme file, apply the "day" one to the stack via `stack.set(DataComponentTypes.ATTRIBUTE_MODIFIERS, ...)`, flag the stack with `NIGHT_STRENGTH` (or a new component if the gate differs), and extend the post-`BONUSES` block in `ArmorEffects` — factor into a helper if/when a second item ever needs it.
- **Attack-triggered custom behavior** (Sage's Grace pattern): add a `ComponentType` in `ModComponents` → `stack.set(...)` it in the theme's `addWeapons` → branch on the flag in `AttackHandlers` (remember to gate side effects to `ServerWorld`, call `stack.damage(...)` yourself since `SUCCESS` skips vanilla wear, and return `ActionResult.SUCCESS` to cancel vanilla damage) → add a tooltip line in `FamSpecialClient`.
- **Attack-triggered additive debuff** (Messmer's Venom pattern): use `ServerLivingEntityEvents.ALLOW_DAMAGE` rather than `AttackEntityCallback` — it fires for *every* damage path (melee, lunge dash, projectile, etc.), while `AttackEntityCallback` only covers left-click melee and misses lunge + projectile hits. Always return `true` from the `ALLOW_DAMAGE` handler (you're adding an effect, not cancelling damage).
- **Block-break hook** (Just Hit Bro pattern): add a `ComponentType` in `ModComponents` → `stack.set(...)` it on the tool in the theme's `addTools` → branch on the flag in `BlockBreakHandler` (gate on `player.getMainHandStack()` to scope to the active hand, wrap side effects in a `ServerWorld` check, pass the tool stack into `Block.dropStacks` so Fortune/Silk Touch apply) → add a cosmetic HUD-badge status effect + `Bonus.effects(...)` entry in `ArmorEffects.BONUSES` gated on the same main-hand flag so badge + gameplay share one source of truth.
- **Custom status effect** (Messmer's Flame Aegis / Messmer's Venom pattern): register in `effect/ModStatusEffects.java` as a `RegistryEntry<StatusEffect>` via `Registry.registerReference(Registries.STATUS_EFFECT, ...)`. For cosmetic HUD badges, use a plain `new StatusEffect(StatusEffectCategory.BENEFICIAL, colorInt) {}` — no overrides. For gameplay-bearing effects, override `canApplyUpdateEffect(int duration, int amplifier)` and `applyUpdateEffect(ServerWorld, LivingEntity, int amplifier)`. Add a lang key at `assets/famspecial/lang/en_us.json` and a vanilla icon at `assets/famspecial/textures/mob_effect/<id>.png` (extract byte-for-byte from the Fabric Loom cache jar — do not author custom art). Apply the effect via a `Bonus.effects(...)` entry in `ArmorEffects.BONUSES` with a trigger that matches the gate of whatever mechanic it's advertising. If the effect needs instant-disappear on trigger drop rather than natural 20s decay, add it to the `MOD_MANAGED` set — vanilla effects like Bad Omen / Haste are deliberately left out so ambient sources (beacons, potions) aren't stripped on unequip.
- **Fixed-loadout / non-modifiable item**: themed gear currently has **no anvil/grindstone lockouts** — the old `HEALS_TARGET`, `BLOCKS_MENDING`+Mending, and `THOR_MACE`+anvil-input blank-output guards were all removed in the anvil/grindstone refactor, and themed gear now behaves exactly like vanilla gear for rename, repair-combine, enchant-merge, and disenchant operations in both creative and survival. If a future item needs a hard lockout, add a dedicated `NO_PLAYER_MOD`-style flag and extend `AnvilScreenHandlerMixin` and/or `GrindstoneScreenHandlerMixin` to check for it. No change needed for the enchanting table (already naturally blocked for pre-enchanted stacks) or smithing table (custom components pass through `SmithingTransformRecipe`).
- **New theme group**: add `item/entries/NewThemeItems.java` with three public `static void` methods — `addWeapons(ItemGroup.Entries entries, RegistryWrapper<Enchantment> enchants)`, `addTools(...)`, `addArmor(...)` — with empty bodies for categories the theme doesn't ship. Add three calls to `ModItemGroups.GEAR_GROUP`'s `.entries(...)` lambda (`NewThemeItems.addWeapons(...)` in the weapons block, `NewThemeItems.addTools(...)` in the tools block) plus one call to `ARMOR_GROUP`'s lambda (`NewThemeItems.addArmor(...)`), all in the desired theme-ordering position.
