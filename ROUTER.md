# ROUTER.md

Task-to-file routing rules. When starting a task in this repo, match it against the rules below and load only the listed topic files. Default to loading fewer files; use `INDEX.md` as a fallback lookup when the task doesn't match any rule cleanly.

Always-loaded: `CLAUDE.md` (skeleton: project facts, working style, commands). Everything below is load-on-demand.

## Routing rules

- **Adding a new item, theme, or set bonus (status-effect half)** → `ITEMS.md`. If the new item touches `ArmorEffects.BONUSES` (any set-bonus status effect, helmet-flag bonus, HUD badge, durability regen, main-hand swap), also load `SYSTEMS.md`. If it adds a new `DataComponentType`, also load `SYSTEMS.md` (for `ModComponents`).
- **Adding attribute modifiers (+max health, speed, armor) to an armor piece** → `ITEMS.md`. No other file needed — values render natively in the vanilla attributes tooltip.
- **Attack-triggered behavior (heal-on-hit, debuff-on-hit, lifesteal, bleed)** → `ITEMS.md` + `SYSTEMS.md` (`AttackHandlers` section). If the effect must replace vanilla damage, use `AttackEntityCallback`; if additive, use `ALLOW_DAMAGE`. See the pattern note in `SYSTEMS.md`.
- **Editing or adding a mixin / gating vanilla screens / durability-cap work / enchant-pipeline interception** → `MIXINS.md`. If the mixin reads a component flag, skim `SYSTEMS.md` (`ModComponents` section) for the component's owner and other consumers.
- **Chestplate→elytra merge / elytra-form themed gear / `IS_FAMSPECIAL_GEAR` identity / merge allowlist work** → `MIXINS.md` (`AnvilScreenHandlerMixin` HEAD-cancellable merge inject, `GrindstoneScreenHandlerMixin` elytra re-stamp) + `SYSTEMS.md` (`ModComponents.copyModComponentsForElytra`, `MOD_COMPONENTS_BASE`, `ELYTRA_EXCLUDED_COMPONENTS`, `ModItemGroups.buildItem` identity stamping) + `ITEMS.md` (per-theme merge notes). A themed diamond chestplate from any of 10 themes (**Mistborn, Pacifist, Poseidon, FireSerpent, Necromancer, KnightRadiant, EshEndraNavesh, Shurima, Thor, RaidersLegacy** — **NOT** OldFam) can be merged with a vanilla elytra in an anvil at a flat cost of 4 levels to produce an elytra that retains the chestplate's identity, custom attributes (minus vanilla armor/toughness), custom name, lore, and custom flags; the merged elytra completes the 4-piece (or 5-piece for Thor) set-bonus check. **Enchant merge**: Mending and Unbreaking entries from **both** the source chestplate and the vanilla elytra carry through to the output, with max level winning when both contribute the same enchant — everything else is stripped. Hardening layers: `IS_FAMSPECIAL_GEAR` identity component (forgery gate), 10-theme allowlist, `copyModComponentsForElytra` excluding `INDESTRUCTIBLE` from the transferred flag set, grindstone re-stamp logic to preserve identity through vanilla disenchant. Adding a new theme to the merge allowlist: edit `MERGE_ELIGIBLE_SETS` in `AnvilScreenHandlerMixin.java`. Adding a new `ComponentType` that should (or should not) transfer across the merge: edit `MOD_COMPONENTS_BASE` and optionally `ELYTRA_EXCLUDED_COMPONENTS` in `ModComponents.java` — missing from the base list means the flag silently won't carry.
- **Grindstone / disenchant behavior on themed gear** → `MIXINS.md` (`GrindstoneScreenHandlerMixin`) + `SYSTEMS.md` (`ModComponents.copyModComponentsForElytra`). All themed gear is grindstone-able per vanilla behavior in both creative and survival; the only mod logic is the merged-elytra identity re-stamp. When a themed elytra is disenchanted, the mixin re-stamps `SET_ID`, `IS_FAMSPECIAL_GEAR`, and the other custom components onto the output via `copyModComponentsForElytra` so the disenchanted elytra keeps its themed identity and set-bonus membership. No blank-output gates on `HEALS_TARGET`, `SET_ID`, or any other flag — those were removed in the recent anvil/grindstone refactor.
- **Instant-apply / instant-strip dispatch (equip hook, slot hook, click-slot / creative-drag hook)** → `MIXINS.md` (`LivingEntityEquipMixin`, `ServerPlayNetworkHandlerMixin`, `ServerPlayNetworkHandlerClickSlotMixin`) + `SYSTEMS.md` (`ArmorEffects.refreshBonusesFor` helper). All three mixins call `ArmorEffects.refreshBonusesFor(player)` TAIL — one entrypoint, covers both APPLY and STRIP via the MOD_MANAGED diff. The 80-tick tick loop is the eventual-consistency safety net.
- **Custom status effect (registration, icon, HUD badge, particle override)** → `SYSTEMS.md` (`ModStatusEffects` section). If the effect is driven by a `Bonus` entry, also read the `ArmorEffects` section in the same file. Icons are byte-for-byte vanilla copies — do NOT author custom art.
- **Changing loot drop rates, tier weights, or map destinations** → `SYSTEMS.md` (Loot injection section). Source of truth for rates is the "Custom Loot Pool Allocation" Google Sheet. Three sibling pools live under `loot/`: `LegendaryPool` (legendary OldFam items + a weight-30 Mending book), `MapsPool` (jungle temple / ancient city / woodland mansion maps), and `ThemedSetsPool` (50 themed-set items + a weight-100 Mending book) — pick the pool matching what's being tuned. For themed-set drop rates / per-tier empty weights / themed item weights → `ThemedSetsPool`. Ancient City map uses a datapack structure tag at `src/main/resources/data/famspecial/tags/worldgen/structure/on_ancient_city_maps.json` — add a sibling tag to support a new map destination. Per saved preference: do NOT add items to loot pools unless explicitly asked.
- **Client-only work (tooltips, HUD overlays, world-space rendering, client-side effects)** → `SYSTEMS.md` (`FamSpecialClient` + `HealthOverlay` sections). Tooltip state must mirror the server-side gate in `ArmorEffects` exactly — use the `grantLine(...)` / `buildSetLine(...)` / `grantLineWithCooldown(...)` helpers.
- **Multiplayer version-mismatch / handshake / login-phase networking** → `SYSTEMS.md` (`VersionHandshake` section). Strict equality policy; bump `mod_version` in `gradle.properties` for releases (`processResources` filters it into `fabric.mod.json`).
- **Necromancer zombie summon behavior (target lock, friendly fire, cleanup, cooldown, sound)** → `SYSTEMS.md` (`NecromancerSummon` section). If touching target-goal swap, also load `MIXINS.md` for `MobEntityAccessor`.
- **Poseidon dry-land riptide behavior** → `MIXINS.md` (`TridentItemMixin`) + `SYSTEMS.md` (`ArmorEffects.hasFullSet` helper). There is **no elytra carve-out** — Poseidon 4/4 dry-land Riptide is explicitly allowed even when the chest slot is a merged elytra (intentional velocity-launcher feature, not an exploit).
- **Knight Radiant Shard durability (indestructible / regen / broken state)** → `MIXINS.md` (the 4 mixins: `ItemStackMixin`, `ItemStackAttributeMixin`, `ItemStackEnchantmentsMixin`, `EnchantmentHelperForEachMixin`) + `SYSTEMS.md` (`ArmorEffects` durability-regen block). Note: `BLOCKS_MENDING` is still stamped on the Shard pieces but no longer enforced anywhere (the anvil and enchanting-table gates that used it were removed); Shard pieces are subject to vanilla Mending behavior at anvil and enchanting table.
- **Thor set behavior (Mjolnir launch, Storm's Awakening, Asgardian's Flight, thunder sound)** → `ITEMS.md` (Thor section) + `SYSTEMS.md` (`MjolnirMaceItem` + `ThorEffects` sections + `ArmorEffects` Thor bonus entries + datapack tag overrides section) + `EFFECTS.md` (Asgardian's Flight / God of Thunder / Storm's Awakening rows). Thor is the only 5/5 set; `ASGARDIANS_FLIGHT` gates Mjolnir's launch via a status-effect predicate rather than re-deriving the set-check. Mjolnir ships with **Riptide III** (`EnchantmentHelper.getTridentSpinAttackStrength` reads the level to scale launch magnitude). Mjolnir is **not** anvil-locked at the mixin layer — it behaves per vanilla mace anvil rules (the TAIL guard that formerly blanked Mjolnir+book anvil outputs has been removed). **Mjolnir's enchant eligibility is "vanilla mace + Riptide only"**, enforced via datapack tag overrides under `data/minecraft/tags/item/enchantable/*.json` (durability / mace / weapon / fire_aspect) plus a vanilla `data/minecraft/enchantment/riptide.json` override — accepts Density / Breach / Wind Burst / Smite / Bane / Fire Aspect / Mending / Unbreaking / Vanishing / Riptide, but **not** Sharpness (gated on `enchantable/sharp_weapon`, deliberately not extended) and **not** Loyalty / Channeling (still gated on the unmodified vanilla `enchantable/trident`). Every successful Mjolnir launch costs **-1 durability** (server-side, at the end of `onStoppedUsing`). Storm's Awakening is gated on **Warrior's Greaves** (the Thor leggings, flagged `TRIGGERS_STORM_AWAKENING`); thunderstorm duration is **2400 t (~2 min)** with an **8%** trigger chance; per-player cooldown is **24000 t (1 MC day)** stamped onto the worn Warrior's Greaves via the `STORM_COOLDOWN_END` component for tooltip MM:SS countdown. Storm Strider boots ship with **Feather Falling IV**. Each Thor armor piece grants **+1 MAX_HEALTH** (+0.5 hearts); full armor set = +2 hearts, and the bonus carries through the chestplate→elytra merge.
- **Modifying which enchantments a custom item can receive at the anvil / enchanting table** → `SYSTEMS.md` (datapack tag overrides section). Mechanism is **datapack tag overrides**, not mixins. Vanilla `Enchantment#isAcceptableItem` checks the enchantment's `supported_items` tag against the registered item id; custom items registered under their own id (e.g. `famspecial:mjolnir`) fall outside any vanilla-id-listed tag. To opt a custom item into a vanilla enchant pool, drop an additive override (`{"replace": false, "values": ["<modid>:<itemid>"]}`) at `data/minecraft/tags/item/enchantable/<pool>.json`. To accept an enchant whose own pool you don't want to fully opt into (e.g. Riptide without Loyalty / Channeling), define a custom `data/<modid>/tags/item/<custom>.json` and drop a vanilla-enchantment override at `data/minecraft/enchantment/<enchant>.json` whose `supported_items` points to your custom tag. Mjolnir is the worked example. The same pattern works in **reverse** to gate a single enchant *out* of a custom item that's otherwise opted into the full pool — Fortune & Glory has an "all crossbow enchants except Multishot" arrangement using exactly this technique: `data/minecraft/enchantment/multishot.json` swaps `supported_items` to a custom tag (`#famspecial:multishot_eligible`) that lists only `minecraft:crossbow`, while Quick Charge / Piercing / Mending / Unbreaking / Vanishing continue reading from the additive `enchantable/*` overrides Fortune & Glory still belongs to. **Adding a tag-override file requires no code changes** — just rebuild and the resource pack ships with the jar.
- **Shurima set behavior (Sun Disc Spear KB-bypass, Sun's Protection boots, Shuriman Endurance 4/4)** → `ITEMS.md` (Shurima section) + `SYSTEMS.md` (`ArmorEffects` Shurima bonus entries: `emperors_divide_mainhand`, `suns_protection_boots_desert`, `shuriman_endurance_full_set_desert`) + `MIXINS.md` (`LivingEntityKnockbackMixin` for Emperor's Divide, `PlayerEntityExhaustionMixin` for Shuriman Endurance, `LivingEntityMixin` ModifyReturnValue #4 for Sun's Protection) + `EFFECTS.md` (Emperor's Divide / Sun's Protection / Shuriman Endurance rows). Both Sun's Protection (boots-alone) and Shuriman Endurance (4/4) gate on a **desert-family biome** — `BiomeKeys.DESERT`, `BADLANDS`, or `ERODED_BADLANDS` (variable name `inDesert` is kept for continuity, semantic is "any desert-family biome"). `SHURIMAN_ENDURANCE` is the only **beacon-style** (non-`MOD_MANAGED`, 4800-t / 4-min, apply-once-gated) custom effect — its trigger drops on biome leave but the timer rides out naturally; do NOT add it to `MOD_MANAGED`. Sun's Protection is **not** in `MOD_MANAGED` — on boots removal or biome leave, the trigger stops re-upping and the effect decays naturally over ~20 s. The equip-event `refreshBonusesFor` call means the re-up stops the same tick as the triggering event, but there is no explicit strip — the lingering damage reduction rides out its remaining duration.
- **Adding a new sound event** → `SYSTEMS.md` (`ModSounds` section). Register event → drop OGG under `assets/famspecial/sounds/` → add `sounds.json` entry.
- **Documentation reorg / CLAUDE.md updates** → edit `CLAUDE.md` + the relevant topic file. Keep `CLAUDE.md` minimal; push detail into topic files.
- **Bumping the mod version for a release** → `CLAUDE.md` (Commands section). Edit `gradle.properties` `mod_version`.
- **Don't know which file applies?** → read `INDEX.md` first; it maps every source file to its topic file.

## Loading discipline

- Do not load all topic files speculatively "in case they're relevant." Pick per this router; load more only if the first pass turns up a cross-reference you actually need.
- If a task crosses categories, load the narrowest pair first (e.g. `ITEMS.md` + `SYSTEMS.md`). Add `MIXINS.md` only if you're actually editing a mixin.
- `CLAUDE.md` "Working style" is binding for all tasks — respect the subagent / parallelism / background-execution triggers.
