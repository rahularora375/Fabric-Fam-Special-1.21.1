# SYSTEMS.md

Non-item architecture: subsystem wiring, components, status effects, sounds, item-group helpers, armor tick effects, attack handlers, Necromancer summon, loot injection, client-side overlays, and networking. Root context: `CLAUDE.md`. For per-theme item content that drives these systems, see `ITEMS.md`. For mixin hooks, see `MIXINS.md`.

Loot-table injection is programmatic — no datapack loot JSONs. The only datapack asset is one structure tag (`data/famspecial/tags/worldgen/structure/on_ancient_city_maps.json`) backing the Ancient City explorer map destination.

## Subsystem wiring

Twelve subsystems registered from `FamSpecial.onInitialize()` in this order: components → status effects → sounds → **ThorItems registration** → item groups → loot modifier → armor effects → attack handlers → block-break handler → Necromancer summon → ThorEffects → version handshake. `ThorItems.register()` runs **before** `ModItemGroups.registerItemGroups()` because registering an `Item` subclass (Mjolnir is `MjolnirMaceItem`, not a vanilla `Item` instance) at static-final class-load crashes in 1.21+ — `SimpleRegistry` rejects intrusive holders during the early-boot window the creative-tab-entries lambda runs in. Additionally, `client/FamSpecialClient` is a **separate `client` entrypoint** declared in `fabric.mod.json`; it runs independently of `onInitialize` and only on the physical client.

**Both-sides install requirement.** The mod must be present on both client and server in multiplayer. `net/VersionHandshake.java` runs a login-phase query on the `famspecial:version` channel: the server sends its `MOD_VERSION` string to every connecting client in `ServerLoginConnectionEvents.QUERY_START`; the client replies with its own version via `ClientLoginNetworking.registerGlobalReceiver`. Mismatch policy is strict equality — any difference, or a client that doesn't understand the query (vanilla / no-mod client), triggers `ServerLoginNetworkHandler.disconnect(...)` with a human-readable message naming both versions. `MOD_VERSION` is read once at class-load from `FabricLoader.getInstance().getModContainer(MOD_ID)` so it always matches the jar's `fabric.mod.json` (populated via `${version}` → `gradle.properties` — see the Commands section for how to bump it for a release). Kicked clients never reach the play state, so there's no partial-state window where loot injection or attack handlers could run for a mismatched peer.

## `component/ModComponents.java`

Registers the custom `DataComponentType`s used to flag behavior on `ItemStack`s (the 1.20.5+ mechanism for attaching state/flags to stacks):

- `famspecial:grants_night_vision` (`Boolean`) — helmet flag driving Night Vision while worn.
- `famspecial:set_id` (`String`) — marks a piece as belonging to a named armor set (currently `"mistborn"`, `"pacifist"`, `"poseidon"`, `"fire_serpent"`, `"necromancer"`, `"knight_radiant"`, `"esh_endra_navesh"`, `"shurima"`, or `"thor"`). Four equipped pieces with matching `SET_ID` trigger that set's bonus (in `ArmorEffects.BONUSES` for status effects, or in `TridentItemMixin` for the Poseidon riptide override). `"thor"` is the only 5/5 set — Mjolnir counts as the 5th piece via its `THOR_MACE` flag, checked separately from the armor-slot `SET_ID` iteration.
- `famspecial:night_strength` (`Boolean`) — main-hand-only flag driving the Atium Dagger's day/night `AttributeModifiersComponent` swap.
- `famspecial:heals_target` (`Boolean`) — main-hand-only flag making attacks heal the target instead of damaging (Sage's Grace). Handled in `AttackHandlers`.
- `famspecial:grants_water_breathing` (`Boolean`) — helmet flag driving Water Breathing while worn (Lorelai's Crown).
- `famspecial:grants_messmers_flame` (`Boolean`) — helmet flag driving Messmer's Flame Aegis (80% fire damage reduction via `LivingEntityMixin`) while worn. Lives on Impaler's Crown; decoupled from the Fire Serpent 4/4 gate so the flame resist can be worn a la carte.
- `famspecial:shows_entity_hp` (`Boolean`) — helmet flag (Sage's Crown) enabling client-side floating HP bars above every remote player in view. Client-only effect, no server behavior; handled in `client/HealthOverlay.java`.
- `famspecial:no_lunge_hunger` (`Boolean`) — weapon flag that suppresses the Lunge enchant's exhaustion cost. Lives on Fire Serpent's Wrath; checked in `ApplyExhaustionEnchantmentEffectMixin` against the enchant context's stack.
- `famspecial:indestructible` (`Boolean`) — elytra-style durability cap + "broken" gate. Enforced in three mixins: `ItemStackMixin` clamps damage at `maxDamage - 1` on the core `damage(...)` overload, plays `ENTITY_ITEM_BREAK` once on the not-broken → broken transition, and calls `player.equipStack(slot, stack)` to refresh the entity's attribute cache. `ItemStackAttributeMixin` HEAD-cancels `applyAttributeModifiers` while the stack is at the cap so the broken piece contributes no armor/toughness/custom attributes. `ItemStackEnchantmentsMixin` returns `ItemEnchantmentsComponent.DEFAULT` from `getEnchantments()` while broken so no enchant benefits (or tooltip entries) apply. `REGENS_DURABILITY` reads bypass both gates (they hit `DataComponentMap` directly), so Stormlight + the 243-tick regen keep firing — regen heals the piece back out of the broken state, at which point `ArmorEffects`' regen block re-invokes `equipStack` to refresh the attribute cache. Lives on the 4 Shard armor pieces.
- `famspecial:regens_durability` (`Boolean`) — passive durability regen flag. Driven by an extra 243-tick cadence block in `ArmorEffects.onEndServerTick` — one durability point restored per slot per 243 ticks while worn. Lives on the 4 Shard armor pieces.
- `famspecial:blocks_mending` (`Boolean`) — Mending-application lockout. Enforced in two places: `AnvilScreenHandlerMixin` blanks the anvil output when the left input is flagged and the right input carries Mending (either as an active enchant or stored enchanted-book entry); `EnchantmentHelperMixin` filters Mending out of `getPossibleEntries` so it can never roll at the enchanting table. Lives on the 4 Shard armor pieces.
- `famspecial:grants_shardbearing` (`Boolean`) — main-hand weapon flag granting **Shardbearing** (+7% of the target's current HP as bonus damage on the wearer's melee hits, applied after all mitigation) while held. Lives on Oathbringer. Two consumers read this flag off the attacker's mainhand stack: `LivingEntityMixin`'s `modifyAppliedDamage` ModifyReturnValue adds the current-HP chip to the final damage; `ArmorEffects`' `shardbearing_mainhand` bonus applies the cosmetic Shardbearing HUD badge. Effect and mixin share one gate so they can't drift.
- `famspecial:grants_undead_resistance` (`Boolean`) — chestplate flag driving 60% damage reduction from undead attackers (melee + ranged). Lives on Smooth Criminal's Vestment (Necromancer chest). Two consumers read the CHEST-slot flag: `LivingEntityMixin`'s `famspecial$undeadResistance` ModifyReturnValue multiplies damage by 0.4× when the attacker's type is in `EntityTypeTags.UNDEAD` (arrow hits attribute back to the shooter via `DamageSource.getAttacker()`, so skeletons + strays are covered by the same branch); `ArmorEffects`' `undead_resistance_chest` bonus applies the cosmetic `UNDEAD_RESISTANCE` HUD badge (display name "Undead Resistance").
- `famspecial:applies_wither_on_hit` (`Boolean`) — main-hand weapon flag that applies Wither II for 4s on any melee hit. Lives on Thriller's Edge (Necromancer axe). Handled additively in `AttackHandlers`' `ALLOW_DAMAGE` handler alongside the Messmer's Venom branch (main-hand only; off-hand flagged stacks don't propagate). `ArmorEffects`' `wither_touch_mainhand` bonus applies the cosmetic `WITHER_TOUCH` HUD badge.
- `famspecial:necromancer_cooldown_end` (`Long`) — absolute world-tick at which the 5-min Necromancer summon cooldown expires. Stamped on every worn Necromancer piece by `NecromancerSummon.stampCooldownOnWornSet` when the summon fires. Client tooltips read this to render the MM:SS countdown next to the "Grants Rotten Muscle" line; server ignores it entirely and keeps using its own per-UUID `lastSummonTick` map (stamping is a display-sync mechanism, not the cooldown source of truth). Stays stamped after expiry — the tooltip just hides once `currentTime >= value`.
- `famspecial:grants_ominous` (`Boolean`) — helmet flag driving Bad Omen while worn at night (timeOfDay ≥ 14000, i.e. 8 PM → 6 AM) or always in Nether/End. Lives on the Esh-Endra-Navesh helmet (Time Dekhlo Helmet). Applied by `ArmorEffects`' `ominous_helmet` bonus. Bad Omen is a vanilla effect, so it's deliberately left out of `MOD_MANAGED` — ambient sources (ominous-bottle, raid captain drops) decay naturally rather than being stripped on unequip.
- `famspecial:bonus_diamond_chance` (`Boolean`) — main-hand pickaxe flag. Every successful block break additionally rolls vanilla's `blocks/diamond_ore` loot table at 1/800, respecting Fortune + Silk Touch because the flagged pickaxe is passed back in as the `TOOL` of the phantom drop. Lives on Just Hit Bro (Esh-Endra-Navesh pickaxe). Handled in `BlockBreakHandler` on `PlayerBlockBreakEvents.AFTER`; advertised via the cosmetic `SHADI_BUFF` HUD badge (`ArmorEffects`' `shadi_buff_mainhand` bonus) while the pickaxe is held.
- `famspecial:ignores_kb_resistance` (`Boolean`) — main-hand weapon flag driving **Emperor's Divide**. Lives on the Sun Disc Spear (Shurima). While the stack is in the mainhand, `ArmorEffects`' `emperors_divide_mainhand` bonus applies `EMPERORS_DIVIDE`; `LivingEntityKnockbackMixin` reads that effect on the attacker in `LivingEntity#takeKnockback` and returns `0.0` from the victim's `KNOCKBACK_RESISTANCE` attribute lookup, so the `(1 - resist)` multiplier becomes 1 and knockback always lands full-strength regardless of target stiffness. Instant removal on scroll/hotbar-key swap via `ServerPlayNetworkHandlerMixin`; MOD_MANAGED fallback catches drag / drop / item break.
- `famspecial:grants_suns_protection` (`Boolean`) — helmet flag driving **Sun's Protection**. Lives on Sun Emperor's Crown (Shurima helmet). Applied by `ArmorEffects`' `suns_protection_helmet_desert` bonus only while the helmet is worn AND the player is in a **desert-family biome** (`BiomeKeys.DESERT`, `BADLANDS`, or `ERODED_BADLANDS`); `LivingEntityMixin`'s `modifyAppliedDamage` reads the effect on the victim and scales incoming damage by 0.8× (Resistance-I-equivalent). **Not in `MOD_MANAGED`** — on helmet remove or biome leave the trigger simply stops re-upping and the effect decays naturally over ~20 s. The `refreshBonusesFor` call from `LivingEntityEquipMixin` does cause the helmet-remove case to stop re-upping within one tick, but there is no explicit strip — the lingering damage reduction rides out the remaining duration in both cases.
- `famspecial:lightning_on_hit` (`Boolean`) — main-hand weapon flag. Lives on Mjolnir. Every successful melee hit rolls 100% (thundering) / 20% (clear) to spawn a `LightningEntity` at the victim. Handled in `ThorEffects.AFTER_DAMAGE`, gated on the attacker's mainhand. `ArmorEffects`' `god_of_thunder_mainhand` bonus applies the cosmetic `GOD_OF_THUNDER` HUD badge so icon + mechanic share one source of truth.
- `famspecial:triggers_storm_awakening` (`Boolean`) — helmet flag. Lives on Thunderhelm. On every kill the wearer performs, roll the Storm's Awakening trigger chance (currently `1.00f` as a TESTING VALUE still flagged as such in code pending ship) to start a thunderstorm — but **only when `world.isThundering()` is false**; if already thundering, the kill is ignored entirely (no roll, no cooldown burn). Storm duration is **4800 t (~4 min real-time)**. Per-player **24000-tick (1 MC day)** cooldown only arms on successful trigger. Handled in `ThorEffects.AFTER_DEATH`. On successful trigger, `STORM_COOLDOWN_END` is stamped onto the worn Thunderhelm for the client tooltip's crimson MM:SS countdown (parallel to the Necromancer cooldown component). `ArmorEffects`' `storms_awakening_helmet_ready` bonus applies the cosmetic `STORMS_AWAKENING` HUD badge while the helmet is worn AND the cooldown is clear — disappears during the 1-MC-day window, re-appears once cleared.
- `famspecial:storm_cooldown_end` (`Long`) — absolute world-tick at which the Storm's Awakening cooldown expires. Stamped on the worn Thunderhelm by `ThorEffects` on a successful storm trigger — parallel to `NECROMANCER_COOLDOWN_END`. Client tooltips read this via `FamSpecialClient.stormCooldownRemaining(ItemStack)` to render the crimson MM:SS countdown next to the "Grants Storm's Awakening" line through `grantLineWithCooldown(...)`. Server ignores this component entirely — the cooldown source of truth is `ThorEffects`' per-UUID map; stamping is a display-sync mechanism.
- `famspecial:thor_mace` (`Boolean`) — marker identifying Mjolnir specifically (not any `MaceItem` instance). Drives the 5th-piece half of the Thor 5/5 gate in `ArmorEffects`' `asgardians_flight_full_thor_set_and_mjolnir` bonus, the right-click riptide launch's weapon check in `MjolnirMaceItem.use` / `onStoppedUsing`, and the mainhand strip check in `ServerPlayNetworkHandlerMixin`. Distinct from `LIGHTNING_ON_HIT` so one flag can in theory be reused on non-Mjolnir Thor weapons without making them count as the 5th piece.

## `effect/ModStatusEffects.java`

Registers custom `StatusEffect` entries as `RegistryEntry<StatusEffect>` values. Two effects carry real gameplay; the rest are cosmetic HUD badges (registered purely so `ArmorEffects.BONUSES` can apply them as top-right icons that truthfully track when their underlying mechanic is firing).

- `MESSMERS_FLAME_AEGIS` (BENEFICIAL) — marker effect checked by `LivingEntityMixin` to scale incoming fire damage to 20%. Driven by the `GRANTS_MESSMERS_FLAME` helmet flag.
- `MESSMERS_VENOM` (BENEFICIAL) — dual-purpose. On the **wearer**, it's a cosmetic aura (the `applyUpdateEffect` override short-circuits damage for players wearing the full Fire Serpent set) plus the trigger `AttackHandlers` reads to decide whether a player's spear hit propagates venom. On **other mobs**, the override mirrors vanilla `PoisonStatusEffect` byte-for-byte — ticks every 25 game-ticks at amp 0, deals 1.0 magic damage, caps at 1 HP. A distinct registry entry from `StatusEffects.POISON` so vanilla `canHaveStatusEffect` equality checks (spiders, undead, the Wither, the Ender Dragon) fall through and the effect applies to every mob.
- `HEALERS_VISION`, `EMPOWERED_RIPTIDE`, `NIGHT_STRENGTH`, `TIRELESS_LUNGE` — cosmetic HUD badges only; no override logic. Colors are tuned to each set's name-prefix palette (cyan / sky-blue / indigo / neon-green). Each is driven by a `Bonus.effects(...)` entry in `ArmorEffects.BONUSES` whose trigger matches the gate of the mechanic it's advertising, so icon and gameplay share one trigger and can't drift. `NIGHT_STRENGTH` is **not in `MOD_MANAGED`** — decays naturally over ~20 s after the gate drops. `NIGHT_STRENGTH`'s icon at `mob_effect/night_strength.png` is now copied from vanilla `wither_rose.png` (previously from vanilla `strength.png`). The remaining three in this row are in `MOD_MANAGED`.
- `STORMLIGHT` (BENEFICIAL, color `0xA0D2FF`) — HUD badge + particle aura for the Knight Radiant Shard set. `applyUpdateEffect` spawns End Rod particles around the wearer every 5 ticks (throttled via `canApplyUpdateEffect`); `canApplyUpdateEffect` is the only non-cosmetic override. The durability-regen mechanic it advertises lives in `ArmorEffects`' 243-tick block, driven by `REGENS_DURABILITY` — the effect itself does not regen anything. Icon copied from vanilla `conduit_power.png`.
- `SHARDBEARING` (BENEFICIAL, color `0xE0F0FF`) — cosmetic HUD badge for Oathbringer. Actual gameplay (+7% of the target's current HP as bonus damage on the wearer's melee hits, applied past armor/protection/resistance) lives in `LivingEntityMixin`'s `modifyAppliedDamage` ModifyReturnValue at `LivingEntityMixin.java:61` (factor `0.07f`), gated on the attacker's mainhand carrying `GRANTS_SHARDBEARING`. Hooked on `modifyAppliedDamage` rather than `applyArmorToDamage` so the bonus lands AFTER all mitigation — matters on unarmored mobs (most of them) where a pre-armor bonus would have been absorbed or eaten. Icon and mixin share the same mainhand gate via `ArmorEffects`' `shardbearing_mainhand` bonus. No time gate. Icon copied from vanilla `item/prismarine_shard.png` (literally a shard — on-theme for the effect name and the Knight Radiant pale-blue palette).
- `RADIANT_MIGHT` (BENEFICIAL, color `0xE0F0FF`) — Knight Radiant 4/4 set bonus. Registers `+2 ATTACK_DAMAGE` (two-thirds of vanilla Strength I's +3.0) as an `ADD_VALUE` modifier via `.addAttributeModifier(...)` chained on registration, so vanilla `StatusEffect` application handles the damage boost automatically — no `applyUpdateEffect` override. Driven by `ArmorEffects`' `knight_radiant_full_set` bonus (all four armor slots share `SET_ID="knight_radiant"`); no time gate. Distinct from the per-piece Stormlight aura. **Not in `MOD_MANAGED`** — decays naturally over ~20 s after the set is broken rather than being diff-stripped within a 4 s cycle. Icon at `mob_effect/radiant_might.png`, copied from vanilla `strength.png`.

## `item/NecromancerSummon.java`

Zombie Reinforcements, the 4/4 Necromancer set bonus. Event-driven (not tick-driven), registers four hooks from `onInitialize`:

1. **`ServerLivingEntityEvents.AFTER_DAMAGE`** — the summon trigger. Fires after mitigation, only when the victim survived, carrying the post-mitigation `DamageSource`. On every hit taken by a 4/4-set-wearer with a living attacker, refresh `latestAttackerBySummoner` (summoner UUID → attacker UUID) and call `retargetActiveZombies` so any summoner's currently-living minions immediately switch to the new hostile — happens regardless of cooldown, so a second attacker during the 5-min window still pulls the existing minions off the first. Then, if not on cooldown, spawn 2 zombies via `spawnZombies`, stamp `NECROMANCER_COOLDOWN_END` onto every worn Necromancer piece (for the client tooltip MM:SS countdown), play `ModSounds.NECROMANCER_SUMMON` on `SoundCategory.HOSTILE`, and instantly strip `ROTTEN_MUSCLE` off the summoner (so the HUD badge flips immediately rather than waiting for the 20s duration to tick out). Also handles the summoner-hits-own-zombie case: if the attacker is the victim's tracked owner, clear the zombie's `setAttacker(null)` + `setTarget(null)` so `RevengeGoal` doesn't latch the summoner — the predicate-gated `ActiveTargetGoal` installed at spawn then re-acquires the original hostile on the next scan.
2. **`ServerLivingEntityEvents.ALLOW_DAMAGE`** — friendly-fire gate. Cancels damage (returns `false`) when the attacker is a tracked zombie AND the victim is that zombie's summoner. Summoner-hits-own-zombie path is NOT cancelled here (the summoner can kill their minions if they choose); the aggro-back is suppressed in the `AFTER_DAMAGE` path above. Load-bearing for blocked / 0-damage hits where `AFTER_DAMAGE` may not fire but `RevengeGoal` still latches.
3. **`ServerLivingEntityEvents.AFTER_DEATH`** — summoner-death cleanup. When a 4/4 wearer dies, iterate `ownerByZombie` and `discard()` every zombie owned by the dying summoner, prune the tracking maps. Leaves `lastSummonTick` + `latestAttackerBySummoner` untouched (harmless without the set equipped, and the player doesn't have the set on respawn anyway).
4. **`ServerTickEvents.END_SERVER_TICK`** — lifetime cleanup (every 20 ticks). Walks `zombieExpireTick` and `discard()`s zombies past their 5-minute expiry. Entries for zombies in unloaded chunks are kept until their expire tick passes — pruning earlier would orphan `setPersistent` zombies whose chunk later reloads (letting a player ferry them into an unloaded area to permanently escape cleanup).

**Spawn loop** mirrors vanilla `ZombieEntity.damage`'s reinforcement mechanic — 50-attempt random-position loop, gated on `SpawnRestriction.isSpawnPosAllowed` + collision (`world.isSpaceEmpty` + `doesNotIntersectEntities`) — but scoped to a tight 3-block horizontal × 1-block vertical radius (vanilla's 7-40 is village-scale; here we want minions beside the wearer). The full `SpawnRestriction.canSpawn` predicate is **intentionally skipped** because it funnels through `HostileEntity.canSpawnInDark`, which fails at sky-light > 7 — i.e. daylight. A 4/4 set bonus must fire regardless of lighting, so position acceptance is ground + collision only.

**Target locking** uses `MobEntityAccessor` (an `@Accessor` mixin widening `MobEntity.targetSelector`) to clear every `ActiveTargetGoal` from the spawned zombie — otherwise vanilla's `ActiveTargetGoal<PlayerEntity>` rescans every tick and retargets the summoner (the nearest player). In their place, install a single predicate-gated `ActiveTargetGoal<LivingEntity>` whose filter reads `latestAttackerBySummoner.get(summonerUuid)` live each tick; updating that map in `onAfterDamage` implicitly re-aims every active zombie for that summoner. `RevengeGoal` is intentionally left in place so non-summoner attackers still draw retaliation.

**Equipment loadout** is built in `equipArmor`: armor is hardcoded (Turtle Helmet + Leather Chestplate with Protection I + Leather Leggings + Leather Boots); the mainhand sword is **per-zombie uniform RNG** chosen from a fixed 6-element `SWORD_TIERS` array — {WOODEN, STONE, IRON, GOLDEN, DIAMOND, NETHERITE} `_SWORD`, unenchanted — via `world.random.nextInt(SWORD_TIERS.length)`. Each of the 2 spawned zombies rolls independently, so a summon can produce e.g. one wooden-sword zombie + one netherite-sword zombie. Drop chance is forced to 0.0 on every slot (armor + mainhand) via `setEquipmentDropChance(slot, 0.0f)` so kills never yield duplicate gear (keeps the summon mechanic from becoming a turtle-helmet or netherite-sword farm). The Protection enchant entry is fetched via `world.getRegistryManager().getOrThrow(RegistryKeys.ENCHANTMENT).getOrThrow(Enchantments.PROTECTION)` and attached via `ItemEnchantmentsComponent.Builder`.

Exposes a public helper `isOnCooldown(PlayerEntity, long worldTime)` consumed by `ArmorEffects`' `rotten_muscle_ready` bonus trigger — so the Rotten Muscle HUD badge gate and the summon gate share one source of truth (the `lastSummonTick` per-UUID map).

## `item/ArmorEffects.java`

Server tick listener (every 80 ticks = 4s, applying 400-tick / 20s effects). The 20s-duration-with-4s-refresh cadence re-ups before the sub-10s vanilla effect blink so buffs stay solid while worn and drop fast on unequip. Three independent mechanisms live here.

**Instant-apply dispatch via `refreshBonusesFor(ServerPlayerEntity player)`.** The body of what used to be the tick-loop iterator has been extracted into a public static helper `ArmorEffects.refreshBonusesFor(player)`: builds a `TickContext` for the player (reads `isNight` off the player's server world), walks `BONUSES`, applies effects, runs the `MOD_MANAGED` diff-eviction, and runs the Obsidian Dagger attribute swap. Idempotent and self-contained. The 80-tick tick loop now just iterates online players and calls `refreshBonusesFor(player)` on each — the slow path's behavior is unchanged and remains the eventual-consistency safety net. Three event-driven mixins call the same helper so the dispatch fires within one tick of the triggering event:

- `LivingEntityEquipMixin` (HEAD on `LivingEntity.onEquipStack`) — covers any armor-slot mutation that funnels through `equipStack` (standard wear, swap, death drop).
- `ServerPlayNetworkHandlerMixin` (TAIL on `onUpdateSelectedSlot`) — covers hotbar scroll / number-key swap.
- `ServerPlayNetworkHandlerClickSlotMixin` (TAIL on `onClickSlot` + `onCreativeInventoryAction`) — covers survival inventory drag / shift-click / armor-swap-key AND creative-tab drag. ## `item/AttackHandlers.java`

Registers two separate hooks because Fabric's two event channels cover different damage paths:

1. **`ServerLivingEntityEvents.ALLOW_DAMAGE`** (pre-damage, fires for *every* damage path since they all funnel through `LivingEntity.damage`) — two additive branches. (a) If the attacker is a player with `MESSMERS_VENOM` and their weapon stack is in `#minecraft:spears`, apply Messmer's Venom (14s, amp 0) to the victim. Gating on the weapon tag (not the set) means melee stabs, lunge dashes, and thrown spear hits all propagate, while fists and non-spear weapons don't — any future modded spear tagged `#minecraft:spears` inherits the behavior for free. (b) If the attacker is a player whose **main-hand** stack carries `APPLIES_WITHER_ON_HIT`, apply Wither II for 4s to the victim. Checked against `player.getMainHandStack()` rather than `source.getWeaponStack()` so off-hand flagged stacks don't propagate — matches the axe's `AttributeModifierSlot.MAINHAND`-gated attack-speed modifier. Always returns `true` — both branches are additive propagation, not cancellations.

2. **`AttackEntityCallback.EVENT`** (pre-damage, left-click melee only — fires inside `PlayerEntity.attack` and misses lunge dashes + projectile hits) — if the attacker's main-hand stack has `HEALS_TARGET`, replicate vanilla's private `isCriticalHit` check + cooldown gate, heal the target (if `LivingEntity`) by 4 HP on a normal hit or 8 HP on a crit, spawn heart particles, play the vanilla attack-sound (STRONG/WEAK/CRIT), consume 3 durability via `stack.damage(...)` (since vanilla's wear path is short-circuited by our cancel), reset the attack cooldown, and return `ActionResult.SUCCESS` to cancel vanilla damage/knockback/sweep/durability. Side effects (heal, sounds, particles, durability) are gated to `ServerWorld` to avoid desync and double-broadcast; the cooldown reset + cancel return run on both sides. Sweep is implicitly defeated — `AttackEntityCallback` never fires for sweep-caught entities, and `SUCCESS` short-circuits `PlayerEntity.attack` before `doSweepingAttack` runs.

Pattern choice: use **`AttackEntityCallback`** for behavior that must *replace* the damage/knockback (heal, freeze, teleport, etc.); use **`ALLOW_DAMAGE`** for behavior that runs *additively* alongside normal damage and needs to cover non-melee paths too (debuffs, bleed, lifesteal).

## `item/BlockBreakHandler.java`

Bonus-diamond drops for `BONUS_DIAMOND_CHANCE`-flagged tools (currently just Just Hit Bro, the Esh-Endra-Navesh pickaxe). Registers one hook on `PlayerBlockBreakEvents.AFTER`, which fires post-break — the block has already been consumed and its own drops have spawned, so the handler layers phantom diamond drops on top rather than replacing anything.

Guards: `ServerWorld` only (AFTER fires client-side too for prediction), server-side `ServerPlayerEntity` only, creative/spectator excluded. Then reads `BONUS_DIAMOND_CHANCE` off the player's **main-hand** stack — so the pickaxe must be the tool that actually broke the block, not just present in the inventory.

Roll: `serverWorld.random.nextInt(800) != 0` early-returns — 1/800 per break. On success, `Block.dropStacks(Blocks.DIAMOND_ORE.getDefaultState(), world, pos, blockEntity=null, player, tool)` runs vanilla's `blocks/diamond_ore` loot table verbatim with the same tool the player used, so **Fortune rolls the raw-diamond multiplier and Silk Touch drops a diamond_ore block** exactly as they would against a real ore break. Then plays `ModSounds.BONUS_DIAMOND` at the block position on `SoundCategory.PLAYERS` (rides the Players volume slider — this is gear-event, not a world-ambient sound), `source=null` so nearby clients also hear the proc.

The roll probability is the single tuning knob, stored as `BONUS_DIAMOND_ROLL_DENOMINATOR` at top of file. Adjust there, not at each call site.

## `item/MjolnirMaceItem.java`

`MaceItem` subclass backing Mjolnir, the Thor signature weapon (and 5th piece of the 5/5 Thor set). Right-click on the main hand starts a `UseAction.SPEAR` charge for `CHARGE_TICKS` (10), then `onStoppedUsing` launches the player along their look vector with vanilla `TridentItem`'s Riptide math. Launch magnitude is looked up dynamically via `EnchantmentHelper.getTridentSpinAttackStrength(stack, player)` at `MjolnirMaceItem.java:74` — the same call vanilla `TridentItem` uses — so the launch **scales with the Riptide enchant level on the stack**; this is **not cosmetic/visual flavor** as earlier docs claimed. Mjolnir ships with **Riptide V** (up from Riptide III), producing a measurably stronger launch than the prior ship value. `ItemEnchantmentsComponent.Builder.set` in `buildItem` writes the level verbatim with **no clamp**, so levels beyond vanilla's III cap persist on the stack and the spin-attack-strength lookup reads them as-is. No water, no rain, no Riptide enchant required to trigger (level 0 still launches — the Riptide level only tunes magnitude). The launch is gated three ways: hand must be `MAIN_HAND`, the stack must carry `THOR_MACE`, and the player must currently have `ASGARDIANS_FLIGHT` — any failure short-circuits to `super.use` / `super.onStoppedUsing` and the mace behaves like a vanilla mace.

**Asgardian's Flight as gameplay predicate.** `MjolnirMaceItem` reads `player.hasStatusEffect(ASGARDIANS_FLIGHT)` rather than re-deriving the 5/5 + mainhand gate at the call site. The effect is the source of truth — applied by `ArmorEffects`' `asgardians_flight_full_thor_set_and_mjolnir` bonus and instant-stripped via `ArmorEffects.refreshBonusesFor(player)` called from the equip / slot / click-slot mixins when either half drops. Decoupling the gate this way means a mid-charge armor/hotbar/inventory swap drops the launch on the next tick without `MjolnirMaceItem` having to re-walk the equipment slots.

**Client-authoritative velocity.** `onStoppedUsing` mirrors vanilla `TridentItem` byte-for-byte — no `if (world.isClient()) return true;` early exit. The launch math must run on **both** client and server: `LocalPlayerEntity` is client-authoritative on its own velocity, so a server-only `addVelocity` becomes an `EntityVelocityUpdateS2CPacket` that the local player ignores, producing spin-without-launch (the v1.2.5 bug). Do not re-introduce the client-side early return. Only the cooldown arming (`getItemCooldownManager().set(stack, RIPTIDE_SPIN_TICKS)`) is server-only — the client receives that via the cooldown packet on the next tick.

## `client/HealthOverlay.java`

Client-only world-space HP bar overlay above remote players. Registered on `WorldRenderEvents.AFTER_ENTITIES`; runs every frame but bails immediately if the local player's HEAD slot isn't flagged `SHOWS_ENTITY_HP` (Sage's Crown). When active, iterates `world.getPlayers()` (narrower/cheaper than full entity iteration), skips self + removed/dead/spectator players + anything beyond `VIEW_DISTANCE_SQ` (48²), and draws a two-quad bar (dark bg + HP-colored fg) above each remaining player's head. Billboards via `matrices.multiply(camera.getRotation())`, scale `0.025` (matches vanilla nameplate scale), head clearance `0.55` blocks above the entity. Uses `RenderLayers.debugQuads()` (POSITION_COLOR pipeline) — text-background layers silently drop writes due to a vertex-format mismatch. Three-band color ramp: green ≥ 66%, yellow ≥ 33%, red otherwise. Purely client-side — health is read live off the tracked client entity, no server involvement.
