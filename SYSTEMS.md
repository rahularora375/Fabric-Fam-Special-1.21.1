# SYSTEMS.md

Non-item architecture: subsystem wiring, components, status effects, sounds, item-group helpers, armor tick effects, attack handlers, Necromancer summon, loot injection, client-side overlays, and networking. Root context: `CLAUDE.md`. For per-theme item content that drives these systems, see `ITEMS.md`. For mixin hooks, see `MIXINS.md`.

Loot-table injection is programmatic — no datapack loot JSONs. Datapack assets in use: one structure tag (`data/famspecial/tags/worldgen/structure/on_ancient_city_maps.json`) backing the Ancient City explorer map destination, plus a small set of enchantment tag overrides driving Mjolnir's enchant eligibility (see "Datapack tag overrides" below).

## Subsystem wiring

Fifteen subsystems registered from `FamSpecial.onInitialize()` in this order: components → status effects → sounds → **ThorItems registration** → **RaidersLegacyItems registration** → item groups → loot modifier → armor effects → attack handlers → block-break handler → Bounty Hunter kills → Necromancer summon → ThorEffects → Fortune & Glory tick handler → version handshake. `ThorItems.register()` and `RaidersLegacyItems.register()` both run **before** `ModItemGroups.registerItemGroups()` because registering an `Item` subclass (Mjolnir is `MjolnirMaceItem`, Fortune & Glory is `FortuneGloryItem`, neither is a vanilla `Item` instance) at static-final class-load crashes in 1.21+ — `SimpleRegistry` rejects intrusive holders during the early-boot window the creative-tab-entries lambda runs in. Additionally, `client/FamSpecialClient` is a **separate `client` entrypoint** declared in `fabric.mod.json`; it runs independently of `onInitialize` and only on the physical client.

**Both-sides install requirement.** The mod must be present on both client and server in multiplayer. `net/VersionHandshake.java` runs a login-phase query on the `famspecial:version` channel: the server sends its `MOD_VERSION` string to every connecting client in `ServerLoginConnectionEvents.QUERY_START`; the client replies with its own version via `ClientLoginNetworking.registerGlobalReceiver`. Mismatch policy is strict equality — any difference, or a client that doesn't understand the query (vanilla / no-mod client), triggers `ServerLoginNetworkHandler.disconnect(...)` with a human-readable message naming both versions. `MOD_VERSION` is read once at class-load from `FabricLoader.getInstance().getModContainer(MOD_ID)` so it always matches the jar's `fabric.mod.json` (populated via `${version}` → `gradle.properties` — see the Commands section for how to bump it for a release). Kicked clients never reach the play state, so there's no partial-state window where loot injection or attack handlers could run for a mismatched peer.

## `component/ModComponents.java`

Registers the custom `DataComponentType`s used to flag behavior on `ItemStack`s (the 1.20.5+ mechanism for attaching state/flags to stacks):

- `famspecial:grants_night_vision` (`Boolean`) — helmet flag driving Night Vision while worn.
- `famspecial:set_id` (`String`) — marks a piece as belonging to a named armor set (currently `"mistborn"`, `"pacifist"`, `"poseidon"`, `"fire_serpent"`, `"necromancer"`, `"knight_radiant"`, `"esh_endra_navesh"`, `"shurima"`, `"thor"`, or `"raider"`). Four equipped pieces with matching `SET_ID` trigger that set's bonus (in `ArmorEffects.BONUSES` for status effects, or in `TridentItemMixin` for the Poseidon riptide override). `"thor"` is the only 5/5 set — Mjolnir counts as the 5th piece via its `THOR_MACE` flag, checked separately from the armor-slot `SET_ID` iteration.
- `famspecial:is_famspecial_gear` (`Boolean`) — generic identity marker stamped on every themed stack produced by `ModItemGroups.buildItem`. Covers all 11 themes (including OldFam, whose pieces don't carry `SET_ID`). Acts as the **NBT-forgery gate** on `AnvilScreenHandlerMixin`'s chestplate→elytra merge: a stack whose `SET_ID` is in the merge allowlist but whose `IS_FAMSPECIAL_GEAR` is missing or not `true` is rejected, so a plain diamond chestplate with a hand-stamped `SET_ID` cannot abuse the merge path. Carried across the merge via `copyModComponentsForElytra`, so the resulting elytra remains flagged.
- `famspecial:night_strength` (`Boolean`) — main-hand-only flag driving the Atium Dagger's day/night `AttributeModifiersComponent` swap.
- `famspecial:heals_target` (`Boolean`) — main-hand-only flag making attacks heal the target instead of damaging (Sage's Grace). Handled in `AttackHandlers`. **Does not gate anvil behavior** — Sage's Grace is anvil-modifiable per vanilla rules; the earlier `AnvilScreenHandlerMixin` TAIL guard that blanked outputs on `HEALS_TARGET` items has been removed.
- `famspecial:grants_water_breathing` (`Boolean`) — helmet flag driving Water Breathing while worn (Lorelai's Crown).
- `famspecial:grants_messmers_flame` (`Boolean`) — helmet flag driving Messmer's Flame Aegis (80% fire damage reduction via `LivingEntityMixin`) while worn. Lives on Impaler's Crown; decoupled from the Fire Serpent 4/4 gate so the flame resist can be worn a la carte.
- `famspecial:shows_entity_hp` (`Boolean`) — helmet flag (Sage's Crown) enabling client-side floating HP bars above every remote player in view. Client-only effect, no server behavior; handled in `client/HealthOverlay.java`.
- `famspecial:no_lunge_hunger` (`Boolean`) — weapon flag that suppresses the Lunge enchant's exhaustion cost. Lives on Fire Serpent's Wrath; checked in `ApplyExhaustionEnchantmentEffectMixin` against the enchant context's stack.
- `famspecial:indestructible` (`Boolean`) — elytra-style durability cap + "broken" gate. Enforced in three mixins: `ItemStackMixin` clamps damage at `maxDamage - 1` on the core `damage(...)` overload, plays `ENTITY_ITEM_BREAK` once on the not-broken → broken transition, and calls `player.equipStack(slot, stack)` to refresh the entity's attribute cache. `ItemStackAttributeMixin` HEAD-cancels `applyAttributeModifiers` while the stack is at the cap so the broken piece contributes no armor/toughness/custom attributes. `ItemStackEnchantmentsMixin` returns `ItemEnchantmentsComponent.DEFAULT` from `getEnchantments()` while broken so no enchant benefits (or tooltip entries) apply. `REGENS_DURABILITY` reads bypass both gates (they hit `DataComponentMap` directly), so Stormlight + the 243-tick regen keep firing — regen heals the piece back out of the broken state, at which point `ArmorEffects`' regen block re-invokes `equipStack` to refresh the attribute cache. Lives on the 4 Shard armor pieces.
- `famspecial:regens_durability` (`Boolean`) — passive durability regen flag. Driven by an extra 243-tick cadence block in `ArmorEffects.onEndServerTick` — one durability point restored per slot per 243 ticks while worn. Lives on the 4 Shard armor pieces.
- `famspecial:blocks_mending` (`Boolean`) — **Currently unused; kept for potential future use.** Historically enforced by an anvil TAIL guard (blanking outputs when the sacrifice carried Mending) and an enchanting-table filter (`EnchantmentHelperMixin.getPossibleEntries`), both of which have been removed. No consumer reads the flag today; it's still stamped on the 4 Shard armor pieces and still copied by `copyModComponentsForElytra` across the chestplate→elytra merge so its presence on merged elytras is preserved for forward compatibility.
- `famspecial:grants_shardbearing` (`Boolean`) — main-hand weapon flag granting **Shardbearing** (+7% of the target's current HP as bonus damage on the wearer's melee hits, applied after all mitigation) while held. Lives on Oathbringer. Two consumers read this flag off the attacker's mainhand stack: `LivingEntityMixin`'s `modifyAppliedDamage` ModifyReturnValue adds the current-HP chip to the final damage; `ArmorEffects`' `shardbearing_mainhand` bonus applies the cosmetic Shardbearing HUD badge. Effect and mixin share one gate so they can't drift.
- `famspecial:grants_undead_resistance` (`Boolean`) — chestplate flag driving 60% damage reduction from undead attackers (melee + ranged). Lives on Smooth Criminal's Vestment (Necromancer chest). Two consumers read the CHEST-slot flag: `LivingEntityMixin`'s `famspecial$undeadResistance` ModifyReturnValue multiplies damage by 0.4× when the attacker's type is in `EntityTypeTags.UNDEAD` (arrow hits attribute back to the shooter via `DamageSource.getAttacker()`, so skeletons + strays are covered by the same branch); `ArmorEffects`' `undead_resistance_chest` bonus applies the cosmetic `UNDEAD_RESISTANCE` HUD badge (display name "Undead Resistance").
- `famspecial:applies_wither_on_hit` (`Boolean`) — main-hand weapon flag that applies Wither II for 4s on any melee hit. Lives on Thriller's Edge (Necromancer axe). Handled additively in `AttackHandlers`' `ALLOW_DAMAGE` handler alongside the Messmer's Venom branch (main-hand only; off-hand flagged stacks don't propagate). `ArmorEffects`' `wither_touch_mainhand` bonus applies the cosmetic `WITHER_TOUCH` HUD badge.
- `famspecial:necromancer_cooldown_end` (`Long`) — absolute world-tick at which the 5-min Necromancer summon cooldown expires. Stamped on every worn Necromancer piece by `NecromancerSummon.stampCooldownOnWornSet` when the summon fires. Client tooltips read this to render the MM:SS countdown next to the "Grants Rotten Muscle" line; server ignores it entirely and keeps using its own per-UUID `lastSummonTick` map (stamping is a display-sync mechanism, not the cooldown source of truth). Stays stamped after expiry — the tooltip just hides once `currentTime >= value`.
- `famspecial:grants_ominous` (`Boolean`) — boots flag driving Bad Omen while worn at night (timeOfDay ≥ 14000, i.e. 8 PM → 6 AM) or always in Nether/End. Lives on the Esh-Endra-Navesh boots (Dhoom Machale). Applied by `ArmorEffects`' `ominous_boots` bonus. Bad Omen is a vanilla effect, so it's deliberately left out of `MOD_MANAGED` — ambient sources (ominous-bottle, raid captain drops) decay naturally rather than being stripped on unequip.
- `famspecial:bonus_diamond_chance` (`Boolean`) — main-hand pickaxe flag. Every successful block break additionally rolls vanilla's `blocks/diamond_ore` loot table at 1/800, respecting Fortune + Silk Touch because the flagged pickaxe is passed back in as the `TOOL` of the phantom drop. Lives on Just Hit Bro (Esh-Endra-Navesh pickaxe). Handled in `BlockBreakHandler` on `PlayerBlockBreakEvents.AFTER`; advertised via the cosmetic `SHADI_BUFF` HUD badge (`ArmorEffects`' `shadi_buff_mainhand` bonus) while the pickaxe is held.
- `famspecial:ignores_kb_resistance` (`Boolean`) — main-hand weapon flag driving **Emperor's Divide**. Lives on the Sun Disc Spear (Shurima). While the stack is in the mainhand, `ArmorEffects`' `emperors_divide_mainhand` bonus applies `EMPERORS_DIVIDE`; `LivingEntityKnockbackMixin` reads that effect on the attacker in `LivingEntity#takeKnockback` and returns `0.0` from the victim's `KNOCKBACK_RESISTANCE` attribute lookup, so the `(1 - resist)` multiplier becomes 1 and knockback always lands full-strength regardless of target stiffness. Instant removal on scroll/hotbar-key swap via `ServerPlayNetworkHandlerMixin`; MOD_MANAGED fallback catches drag / drop / item break.
- `famspecial:grants_suns_protection` (`Boolean`) — boots flag driving **Sun's Protection**. Lives on Drifting Sands (Shurima boots). Applied by `ArmorEffects`' `suns_protection_boots_desert` bonus only while the boots are worn AND the player is in a **desert-family biome** (`BiomeKeys.DESERT`, `BADLANDS`, or `ERODED_BADLANDS`); `LivingEntityMixin`'s `modifyAppliedDamage` reads the effect on the victim and scales incoming damage by 0.8× (Resistance-I-equivalent). **Not in `MOD_MANAGED`** — on boots remove or biome leave the trigger simply stops re-upping and the effect decays naturally over ~20 s. The `refreshBonusesFor` call from `LivingEntityEquipMixin` does cause the boots-remove case to stop re-upping within one tick, but there is no explicit strip — the lingering damage reduction rides out the remaining duration in both cases.
- `famspecial:lightning_on_hit` (`Boolean`) — main-hand weapon flag. Lives on Mjolnir. Every successful melee hit rolls 100% (thundering) / 20% (clear) to spawn a `LightningEntity` at the victim. Handled in `ThorEffects.AFTER_DAMAGE`, gated on the attacker's mainhand. `ArmorEffects`' `god_of_thunder_mainhand` bonus applies the cosmetic `GOD_OF_THUNDER` HUD badge so icon + mechanic share one source of truth.
- `famspecial:triggers_storm_awakening` (`Boolean`) — leggings flag. Lives on Warrior's Greaves (Thor leggings). On every kill the wearer performs, roll the Storm's Awakening trigger chance (**flat 8%**, `STORM_TRIGGER_CHANCE = 0.08f`) to start a thunderstorm — but **only when `world.isThundering()` is false**; if already thundering, the kill is ignored entirely (no roll, no cooldown burn). Storm duration is **2400 t (~2 min real-time)**. Per-player **24000-tick (1 MC day)** cooldown only arms on successful trigger. Handled in `ThorEffects.AFTER_DEATH`. On successful trigger, `STORM_COOLDOWN_END` is stamped onto the worn Warrior's Greaves for the client tooltip's crimson MM:SS countdown (parallel to the Necromancer cooldown component). `ArmorEffects`' `storms_awakening_legs_ready` bonus applies the cosmetic `STORMS_AWAKENING` HUD badge while the leggings are worn AND the cooldown is clear — disappears during the 1-MC-day window, re-appears once cleared.
- `famspecial:storm_cooldown_end` (`Long`) — absolute world-tick at which the Storm's Awakening cooldown expires. Stamped on the worn Warrior's Greaves by `ThorEffects` on a successful storm trigger — parallel to `NECROMANCER_COOLDOWN_END`. Client tooltips read this via `FamSpecialClient.stormCooldownRemaining(ItemStack)` to render the crimson MM:SS countdown next to the "Grants Storm's Awakening" line through `grantLineWithCooldown(...)`. Server ignores this component entirely — the cooldown source of truth is `ThorEffects`' per-UUID map; stamping is a display-sync mechanism.
- `famspecial:thor_mace` (`Boolean`) — marker identifying Mjolnir specifically (not any `MaceItem` instance). Drives the 5th-piece half of the Thor 5/5 gate in `ArmorEffects`' `asgardians_flight_full_thor_set_and_mjolnir` bonus, the right-click riptide launch's weapon check in `MjolnirMaceItem.use` / `onStoppedUsing`, and the mainhand strip check in `ServerPlayNetworkHandlerMixin`. Distinct from `LIGHTNING_ON_HIT` so one flag can in theory be reused on non-Mjolnir Thor weapons without making them count as the 5th piece. **Does not gate anvil behavior** — Mjolnir is anvil-modifiable per vanilla mace rules; the earlier `AnvilScreenHandlerMixin` TAIL guard that blanked Mjolnir+book anvil outputs has been removed.
- `famspecial:bounty_hunter` (`Boolean`) — leggings flag. Lives on Trousers of the Trail (Raider's Legacy legs). Drives the **Bounty Hunter** mechanic in `BountyHunterKills` — when the wearer kills a mob with a vanilla bow/crossbow arrow (incl. spectral arrows; firework rockets excluded by the `ArrowEntity` predicate), 5% chance to drop 1–3 diamonds at the corpse and play `ModSounds.RAJ_TOTEM`. Also drives `ArmorEffects`' `bounty_hunter_legs` bonus, which applies the cosmetic `BOUNTY_HUNTER` HUD badge while the leggings are worn so badge + mechanic share one source of truth. Appended to `MOD_COMPONENTS_BASE` so it carries through any future component-copy path.
- `famspecial:crusaders_volley` (`Boolean`) — main-hand crossbow flag. Lives on Fortune & Glory (Raider's Legacy crossbow). Read by `FortuneGloryItem.onStoppedUsing` to schedule the 2-tick- and 4-tick-delayed follow-up shots, and by `ArmorEffects`' `crusaders_volley_mainhand` bonus to apply the cosmetic `CRUSADERS_VOLLEY` HUD badge while the crossbow is held. Follow-up shots inherit Multishot/Piercing levels from the source stack at fire time. Appended to `MOD_COMPONENTS_BASE`.

**Component-copy helpers (`copyModComponents` / `copyModComponentsForElytra`).** The file exposes two static copy methods that transplant the mod's custom components from a source stack to a destination stack. Both iterate over the shared constant `MOD_COMPONENTS_BASE` — the canonical allowlist of every themed `ComponentType` that should carry across a stack merge. `copyModComponents(from, to)` carries the full list (reserved for armor-to-armor transplant paths; not used by the current chestplate→elytra merge). `copyModComponentsForElytra(from, to)` is the variant invoked by `AnvilScreenHandlerMixin.famspecial$buildMergedElytra` and `GrindstoneScreenHandlerMixin.famspecial$restampMergedElytra` — it skips any entry listed in `ELYTRA_EXCLUDED_COMPONENTS` (currently just `INDESTRUCTIBLE`). Rationale: `INDESTRUCTIBLE` is redundant on elytra because vanilla already caps elytra damage at `maxDamage - 1` natively (the `ItemStackMixin` comment block that documents our cap is explicitly modeled after that vanilla elytra behavior), and carrying it over would interact badly with the broken-state enchant-hiding mixins. `REGENS_DURABILITY` **is** intentionally transferred — the merged elytra keeps the source theme's regen-managed durability design. `BLOCKS_MENDING` is also transferred, but is currently a no-op flag (no consumer reads it) — kept in the copy path for forward compatibility.

**Discipline for new components.** Any new `ComponentType` added to this file MUST be appended to `MOD_COMPONENTS_BASE` or it won't carry across any merge path. If the new component shouldn't follow the chestplate→elytra merge (e.g. it's semantically slot-specific or would enable an exploit on the elytra slot), also add it to `ELYTRA_EXCLUDED_COMPONENTS`.

## `effect/ModStatusEffects.java`

Registers custom `StatusEffect` entries as `RegistryEntry<StatusEffect>` values. Two effects carry real gameplay; the rest are cosmetic HUD badges (registered purely so `ArmorEffects.BONUSES` can apply them as top-right icons that truthfully track when their underlying mechanic is firing).

- `MESSMERS_FLAME_AEGIS` (BENEFICIAL) — marker effect checked by `LivingEntityMixin` to scale incoming fire damage to 20%. Driven by the `GRANTS_MESSMERS_FLAME` helmet flag.
- `MESSMERS_VENOM` (BENEFICIAL) — dual-purpose. On the **wearer**, it's a cosmetic aura (the `applyUpdateEffect` override short-circuits damage for players wearing the full Fire Serpent set) plus the trigger `AttackHandlers` reads to decide whether a player's spear hit propagates venom. On **other mobs**, the override mirrors vanilla `PoisonStatusEffect` byte-for-byte — ticks every 25 game-ticks at amp 0, deals 1.0 magic damage, caps at 1 HP. A distinct registry entry from `StatusEffects.POISON` so vanilla `canHaveStatusEffect` equality checks (spiders, undead, the Wither, the Ender Dragon) fall through and the effect applies to every mob.
- `HEALERS_VISION`, `EMPOWERED_RIPTIDE`, `NIGHT_STRENGTH`, `TIRELESS_LUNGE` — cosmetic HUD badges only; no override logic. Colors are tuned to each set's name-prefix palette (cyan / sky-blue / indigo / neon-green). Each is driven by a `Bonus.effects(...)` entry in `ArmorEffects.BONUSES` whose trigger matches the gate of the mechanic it's advertising, so icon and gameplay share one trigger and can't drift. `NIGHT_STRENGTH` is **not in `MOD_MANAGED`** — decays naturally over ~20 s after the gate drops. `NIGHT_STRENGTH`'s icon at `mob_effect/night_strength.png` is now copied from vanilla `wither_rose.png` (previously from vanilla `strength.png`). The remaining three in this row are in `MOD_MANAGED`.
- `STORMLIGHT` (BENEFICIAL, color `0xA0D2FF`) — HUD badge + particle aura for the Knight Radiant Shard set. `applyUpdateEffect` spawns End Rod particles around the wearer every 5 ticks (throttled via `canApplyUpdateEffect`); `canApplyUpdateEffect` is the only non-cosmetic override. The durability-regen mechanic it advertises lives in `ArmorEffects`' 243-tick block, driven by `REGENS_DURABILITY` — the effect itself does not regen anything. Icon copied from vanilla `conduit_power.png`.
- `SHARDBEARING` (BENEFICIAL, color `0xE0F0FF`) — cosmetic HUD badge for Oathbringer. Actual gameplay (+7% of the target's current HP as bonus damage on the wearer's melee hits, applied past armor/protection/resistance) lives in `LivingEntityMixin`'s `modifyAppliedDamage` ModifyReturnValue at `LivingEntityMixin.java:61` (factor `0.07f`), gated on the attacker's mainhand carrying `GRANTS_SHARDBEARING`. Hooked on `modifyAppliedDamage` rather than `applyArmorToDamage` so the bonus lands AFTER all mitigation — matters on unarmored mobs (most of them) where a pre-armor bonus would have been absorbed or eaten. Icon and mixin share the same mainhand gate via `ArmorEffects`' `shardbearing_mainhand` bonus. No time gate. Icon copied from vanilla `item/prismarine_shard.png` (literally a shard — on-theme for the effect name and the Knight Radiant pale-blue palette).
- `RADIANT_MIGHT` (BENEFICIAL, color `0xE0F0FF`) — Knight Radiant 4/4 set bonus. Registers `+2 ATTACK_DAMAGE` (two-thirds of vanilla Strength I's +3.0) as an `ADD_VALUE` modifier via `.addAttributeModifier(...)` chained on registration, so vanilla `StatusEffect` application handles the damage boost automatically — no `applyUpdateEffect` override. Driven by `ArmorEffects`' `knight_radiant_full_set` bonus (all four armor slots share `SET_ID="knight_radiant"`); no time gate. Distinct from the per-piece Stormlight aura. **Not in `MOD_MANAGED`** — decays naturally over ~20 s after the set is broken rather than being diff-stripped within a 4 s cycle. Icon at `mob_effect/radiant_might.png`, copied from vanilla `strength.png`.
- `ROTTEN_MUSCLE` (BENEFICIAL, color `0x8B0000`, display name "Rotten Muscle") — Necromancer 4/4 HUD badge advertising the Zombie-Reinforcements charge state. Applied by `ArmorEffects`' `rotten_muscle_ready` bonus only while the full set is worn AND `NecromancerSummon.isOnCooldown(...)` returns `false` — so the badge appears when the summon is charged and disappears during the 5-minute cooldown window, then re-appears when the cooldown elapses. On successful summon, `NecromancerSummon` calls `player.removeStatusEffect(ROTTEN_MUSCLE)` to flip the HUD instantly instead of waiting for the 20s duration to tick out. `canApplyUpdateEffect` + `applyUpdateEffect` spawn a single `ParticleTypes.SOUL` wisp around the wearer every 10 ticks — the "necromantic aura" visual cue that the summon is primed. Applied in `ArmorEffects` with `showParticles=false` so the vanilla ambient bubbles don't overlap the themed soul wisps. Icon copied byte-for-byte from vanilla `mob_effect/wither.png`.
- `UNDEAD_RESISTANCE` (BENEFICIAL, color `0x6B8E5A`, display name "Undead Resistance") — Necromancer chestplate (Smooth Criminal's Vestment) HUD badge + particle aura. Advertises the 60% undead-damage reduction. The reduction itself lives in `LivingEntityMixin`, gated on the CHEST-slot `GRANTS_UNDEAD_RESISTANCE` component — effect and mixin share one flag so they can't drift. `canApplyUpdateEffect` + `applyUpdateEffect` spawn `LARGE_SMOKE` every 12 ticks (the one Necromancer effect that still radiates visible particles — the green smoke is the visual cue for the piece-alone bonus). ## `item/ModItemGroups.java`

Registration hub + shared helpers, not the item catalog. Registers **two** sibling creative tabs:

- **`GEAR_GROUP`** (id `famspecial:gear`, icon = `IRON_PICKAXE`, lang key `itemgroup.famspecial.gear`). Its `.entries(...)` lambda calls every theme's `addWeapons(entries, enchants)` first (all weapons, grouped by type across themes), then every theme's `addTools(entries, enchants)`. Each theme file exposes three methods (`addWeapons` / `addTools` / `addArmor`) with empty bodies for categories that theme doesn't ship — e.g. EshEndraNavesh has no weapons, Mistborn has no tools. Within a theme the order is vanilla-page-aligned: weapons = sword(s) → axe → bow → trident; tools = pickaxe → shovel → hoe; armor = helmet → chest → legs → boots.

Exposes public helpers consumed by the `entries/` subpackage:

- `buildItem(item, name, lore, enchants, EnchantEntry...)` — applies `CUSTOM_NAME`, `LORE`, and `ENCHANTMENTS` components to a fresh stack. Also stamps `IS_FAMSPECIAL_GEAR=true` on every stack it produces (immediately after the `CUSTOM_NAME` / `LORE` writes, at `ModItemGroups.java:132`). This is the single chokepoint covering all themed items across all 11 theme groups (including OldFam) — weapons, tools, and armor all route through this helper, so the identity flag propagates to creative-tab stacks and to loot-injected stacks built via `LootPoolHelpers.namedItem(...)` which mirrors the same stamp. The stamp lives at the per-stack layer, **not** the `Item.Settings` default-component layer, because `ModItemGroups` reuses vanilla item instances (`Items.DIAMOND_CHESTPLATE`, etc.) rather than registering custom `Item` subclasses with a `Settings` builder — the per-stack layer is the only universal layer for this codebase's item-registration shape.
- `loreLine(base, segments...)` — mixes inline `String`/`Text` segments on a shared base style so a single lore line can have accent-styled words.
- `e(key, level)` / `EnchantEntry` — fixed-level enchants for creative-tab display (loot-table enchants use ranges — see `LootPoolHelpers`).

Items with passive effects set their component flags on the stack here before adding, e.g. `stack.set(ModComponents.GRANTS_NIGHT_VISION, true)`.

## `entity/ModEntities.java` and `entity/VolleyArrowEntity.java`

Custom `EntityType` registry plus the single custom entity that uses it. Wired from `FamSpecial.onInitialize` via `ModEntities.register()` (early init, alongside `ModSounds.register()` / `ModComponents.register()`) and from `FamSpecialClient.onInitializeClient` via `EntityRendererRegistry.register(ModEntities.VOLLEY_ARROW, ArrowEntityRenderer::new)` (the renderer is reused verbatim from vanilla — no custom texture or model).

`ModEntities.VOLLEY_ARROW` is built via `EntityType.Builder.create(VolleyArrowEntity::new, SpawnGroup.MISC).dropsNothing().dimensions(0.5F, 0.5F).eyeHeight(0.13F).maxTrackingRange(4).trackingTickInterval(20).build(VOLLEY_ARROW_KEY)` — these settings mirror vanilla `EntityType.ARROW` exactly so volley arrows track and serialize to the client identically to a regular arrow.

`VolleyArrowEntity` extends `ArrowEntity` and exists for one reason: bypass the target's invulnerability frames on hit. Vanilla `LivingEntity.timeUntilRegen` defaults to 10 ticks after a hit, and same-or-lesser damage during that window is silently rejected. The 2-tick cadence between Crusader's Volley shots falls inside that window, so shots 2 and 3 would not damage without bypass. `VolleyArrowEntity.onEntityHit(EntityHitResult)` zeroes `target.timeUntilRegen` when the hit is a `LivingEntity`, then delegates to `super.onEntityHit(...)` so vanilla's damage logic runs against a clean iframe state. Nothing else about hit behavior is overridden — Multishot/Piercing inheritance, damage scaling, knockback, all stay vanilla.

Two constructors. The deserialization-side `VolleyArrowEntity(EntityType<? extends VolleyArrowEntity>, World)` is required by the entity-spawn pipeline (client receives a spawn packet for the custom EntityType, looks up the factory, calls this ctor). The convenience-side `VolleyArrowEntity(World, LivingEntity owner, ItemStack arrowStack, ItemStack crossbowStack)` is what `FortuneGloryItem.fireScheduledShot` uses — it can't simply chain through `ArrowEntity`'s convenience constructor because that hardcodes `EntityType.ARROW` and would make the spawn packet say ARROW (defeating the custom-entity-on-client purpose). Instead it chains through the deserialization super, then manually replicates `PersistentProjectileEntity`'s convenience-ctor body: `setStack(arrowStack.copy())`, `copyComponentsFrom(arrowStack)`, INTANGIBLE_PROJECTILE pickup-permission handling, `setPosition(...)`, then stashes the crossbow stack into a private `weaponStack` field and computes Piercing into a private `cachedPierceLevel` byte (the parent's `weapon` field and `setPierceLevel` are private with no setters, so `getWeaponStack()` and `getPierceLevel()` are overridden to return the stashed values). Damage-scaling enchant lookup (Power, Smite-on-arrow, etc.) reads through the overridden `getWeaponStack()`; piercing reads through the overridden `getPierceLevel()`. Net effect: vanilla behavior for everything except the iframe field.

## `item/NecromancerSummon.java`

Zombie Reinforcements, the 4/4 Necromancer set bonus. Event-driven (not tick-driven), registers four hooks from `onInitialize`:

1. **`ServerLivingEntityEvents.AFTER_DAMAGE`** — the summon trigger. Fires after mitigation, only when the victim survived, carrying the post-mitigation `DamageSource`. On every hit taken by a 4/4-set-wearer with a living attacker, refresh `latestAttackerBySummoner` (summoner UUID → attacker UUID) and call `retargetActiveZombies` so any summoner's currently-living minions immediately switch to the new hostile — happens regardless of cooldown, so a second attacker during the 5-min window still pulls the existing minions off the first. Then, if not on cooldown, spawn 2 zombies via `spawnZombies`, stamp `NECROMANCER_COOLDOWN_END` onto every worn Necromancer piece (for the client tooltip MM:SS countdown), play `ModSounds.NECROMANCER_SUMMON` on `SoundCategory.HOSTILE`, and instantly strip `ROTTEN_MUSCLE` off the summoner (so the HUD badge flips immediately rather than waiting for the 20s duration to tick out). Also handles the summoner-hits-own-zombie case: if the attacker is the victim's tracked owner, clear the zombie's `setAttacker(null)` + `setTarget(null)` so `RevengeGoal` doesn't latch the summoner — the predicate-gated `ActiveTargetGoal` installed at spawn then re-acquires the original hostile on the next scan.
2. **`ServerLivingEntityEvents.ALLOW_DAMAGE`** — friendly-fire gate. Cancels damage (returns `false`) when the attacker is a tracked zombie AND the victim is that zombie's summoner. Summoner-hits-own-zombie path is NOT cancelled here (the summoner can kill their minions if they choose); the aggro-back is suppressed in the `AFTER_DAMAGE` path above. Load-bearing for blocked / 0-damage hits where `AFTER_DAMAGE` may not fire but `RevengeGoal` still latches.
3. **`ServerLivingEntityEvents.AFTER_DEATH`** — summoner-death cleanup. When a 4/4 wearer dies, iterate `ownerByZombie` and `discard()` every zombie owned by the dying summoner, prune the tracking maps. Leaves `lastSummonTick` + `latestAttackerBySummoner` untouched (harmless without the set equipped, and the player doesn't have the set on respawn anyway).
4. **`ServerTickEvents.END_SERVER_TICK`** — lifetime cleanup (every 20 ticks). Walks `zombieExpireTick` and `discard()`s zombies past their 5-minute expiry. Entries for zombies in unloaded chunks are kept until their expire tick passes — pruning earlier would orphan `setPersistent` zombies whose chunk later reloads (letting a player ferry them into an unloaded area to permanently escape cleanup).

**Spawn loop** mirrors vanilla `ZombieEntity.damage`'s reinforcement mechanic — 50-attempt random-position loop, gated on `SpawnRestriction.isSpawnPosAllowed` + collision (`world.isSpaceEmpty` + `doesNotIntersectEntities`) — but scoped to a tight 3-block horizontal × 1-block vertical radius (vanilla's 7-40 is village-scale; here we want minions beside the wearer). The full `SpawnRestriction.canSpawn` predicate is **intentionally skipped** because it funnels through `HostileEntity.canSpawnInDark`, which fails at sky-light > 7 — i.e. daylight. A 4/4 set bonus must fire regardless of lighting, so position acceptance is ground + collision only.

**Target locking** uses `MobEntityAccessor` (an `@Accessor` mixin widening `MobEntity.targetSelector`) to clear every `ActiveTargetGoal` from the spawned zombie — otherwise vanilla's `ActiveTargetGoal<PlayerEntity>` rescans every tick and retargets the summoner (the nearest player). In their place, install a single predicate-gated `ActiveTargetGoal<LivingEntity>` whose filter reads `latestAttackerBySummoner.get(summonerUuid)` live each tick; updating that map in `onAfterDamage` implicitly re-aims every active zombie for that summoner. `RevengeGoal` is intentionally left in place so non-summoner attackers still draw retaliation.

**Equipment loadout** is built in `equipArmor`: armor is hardcoded (Turtle Helmet + Leather Chestplate with Protection I + Leather Leggings + Leather Boots); the mainhand sword is **per-zombie uniform RNG** chosen from a fixed 6-element `SWORD_TIERS` array — {WOODEN, STONE, IRON, GOLDEN, DIAMOND, NETHERITE} `_SWORD`, unenchanted — via `world.random.nextInt(SWORD_TIERS.length)`. Each of the 2 spawned zombies rolls independently, so a summon can produce e.g. one wooden-sword zombie + one netherite-sword zombie. Drop chance is forced to 0.0 on every slot (armor + mainhand) via `setEquipmentDropChance(slot, 0.0f)` so kills never yield duplicate gear (keeps the summon mechanic from becoming a turtle-helmet or netherite-sword farm). The Protection enchant entry is fetched via `world.getRegistryManager().getOrThrow(RegistryKeys.ENCHANTMENT).getOrThrow(Enchantments.PROTECTION)` and attached via `ItemEnchantmentsComponent.Builder`.

Exposes a public helper `isOnCooldown(PlayerEntity, long worldTime)` consumed by `ArmorEffects`' `rotten_muscle_ready` bonus trigger — so the Rotten Muscle HUD badge gate and the summon gate share one source of truth (the `lastSummonTick` per-UUID map).

## Loot injection

**`loot/ModLootTableModifier.java`** — hooks Fabric's `LootTableEvents.MODIFY` and delegates to `LegendaryPool.buildFor`, `MapsPool.buildFor`, and `ThemedSetsPool.buildFor`, appending whichever pools they return to the vanilla chest's loot table. Source of truth for drop rates is the "Custom Loot Pool Allocation" Google Sheet.

**`loot/LegendaryPool.java`** — builds the legendary-items pool across four tiers (`CRAZY_TIER`, `S_TIER`, `A_TIER`, `F_TIER`) as `Set<RegistryKey<LootTable>>` of vanilla `LootTables` constants. Per-tier `emptyWeight`s: **Crazy=640, S=1440, A=1840, F=3040**. Item weights are 10 per legendary (×13) plus 30 for the weight-30 Mending book entry = **160** total item weight. With current empty weights that yields drop rates of ~20% / 10% / 8% / 5% (Crazy/S/A/F) — **the Google Sheet is authoritative if these drift**. The `namedItem(...)` helper in `LootPoolHelpers` is parallel to `buildItem` but uses loot functions (`SetNameLootFunction`, `SetLoreLootFunction`, `SetEnchantmentsLootFunction`) and supports enchantment **ranges** via `UniformLootNumberProvider`. The Mending book is a regular `ItemEntry.builder(Items.BOOK).weight(30)` entry — not guaranteed; it competes with the legendaries and the empty entry on the same weighted roll.

**`loot/MapsPool.java`** — builds a separate maps pool across the same four tiers. Three maps (jungle temple, ancient city, woodland mansion) each at weight 20 with per-tier empty weights **Crazy=5,999,940, S=5,999,940, A=340, F=540**, targeting overall map drop rates of ~0% / ~0% / 15% / 10%. Crazy/S empty weights are intentionally huge — map entries are kept structurally present so the sheet's tier schema stays uniform but effectively never roll. Ancient City map uses the custom `famspecial:on_ancient_city_maps` structure tag — the tag JSON lives at `src/main/resources/data/famspecial/tags/worldgen/structure/on_ancient_city_maps.json`; add a sibling file under the same path to support a new map destination. Non-overworld chests (End City, Nether Bridge, Bastions) are commented out of the map tier sets because `ExplorationMapLootFunction` searches in the chest's dimension and would silently produce blank maps.

**`loot/ThemedSetsPool.java`** — builds a themed-gear pool across the same four tiers as `LegendaryPool` (reuses `LegendaryPool.CRAZY_TIER` / `S_TIER` / `A_TIER` / `F_TIER` for the chest-id mapping). Injects **50 themed items** — 5 pieces from each of the 10 non-OldFam themes (Mistborn, Pacifist, Poseidon, FireSerpent, Necromancer, KnightRadiant, EshEndraNavesh, Shurima, Thor, RaidersLegacy) — each at **weight 10**, plus a Mending book at **weight 100**. Per-tier empty-slot weights: **Crazy=1900, S=4400, A=9400, F=19400**. Total item weight per tier = 50 × 10 + 100 = **600**. **OldFam is intentionally excluded** because OldFam items already drop via `LegendaryPool` and would otherwise double-roll. Mechanism: each themed item is wrapped in a `PrebuiltStackEntry` whose `Supplier<ItemStack>` calls the corresponding `build<ItemName>(RegistryWrapper<Enchantment>)` factory in the theme's `entries/*Items.java` — so the chest stack matches the creative-tab stack byte-for-byte (same components, same enchants, same custom name/lore). See "Per-item factory pattern" cross-ref in `ITEMS.md`'s entries-file descriptions; the `addWeapons` / `addTools` / `addArmor` lambdas now thinly delegate to those public factories.

**`loot/PrebuiltStackEntry.java`** — custom `LootPoolEntryType` registered as `famspecial:prebuilt_stack`. Subclass of `LeafEntry` that wraps a `Supplier<ItemStack>` and emits `supplier.get().copy()` on each roll, so every chest spawn is a fresh, independent stack (mutating one drop never aliases another). Registered via `PrebuiltStackEntry.register()` called from `FamSpecial#onInitialize`. Used exclusively by `ThemedSetsPool` to keep the chest output identical to the per-item factory output without re-implementing the build logic in loot-function form.

## `item/ArmorEffects.java`

Server tick listener (every 80 ticks = 4s, applying 400-tick / 20s effects). The 20s-duration-with-4s-refresh cadence re-ups before the sub-10s vanilla effect blink so buffs stay solid while worn and drop fast on unequip. Three independent mechanisms live here.

**Instant-apply dispatch via `refreshBonusesFor(ServerPlayerEntity player)`.** The body of what used to be the tick-loop iterator has been extracted into a public static helper `ArmorEffects.refreshBonusesFor(player)`: builds a `TickContext` for the player (reads `isNight` off the player's server world), walks `BONUSES`, applies effects, runs the `MOD_MANAGED` diff-eviction, and runs the Obsidian Dagger attribute swap. Idempotent and self-contained. The 80-tick tick loop now just iterates online players and calls `refreshBonusesFor(player)` on each — the slow path's behavior is unchanged and remains the eventual-consistency safety net. Three event-driven mixins call the same helper so the dispatch fires within one tick of the triggering event:

- `LivingEntityEquipMixin` (HEAD on `LivingEntity.onEquipStack`) — covers any armor-slot mutation that funnels through `equipStack` (standard wear, swap, death drop).
- `ServerPlayNetworkHandlerMixin` (TAIL on `onUpdateSelectedSlot`) — covers hotbar scroll / number-key swap.
- `ServerPlayNetworkHandlerClickSlotMixin` (TAIL on `onClickSlot` + `onCreativeInventoryAction`) — covers survival inventory drag / shift-click / armor-swap-key AND creative-tab drag. The creative hook is load-bearing because creative-tab armor drag does not route through `onClickSlot`; without it, HUD icons for armor-driven effects lagged up to 4 s in creative.

The refactor collapsed the mixins' previous hand-rolled enumerated strip lists (eight entries in `LivingEntityEquipMixin`, six in `ServerPlayNetworkHandlerMixin`) down to a single `refreshBonusesFor` call apiece — both APPLY and STRIP now flow through the MOD_MANAGED diff in one place. **Result**: six previously-effect-gated gameplay paths (`EMPERORS_DIVIDE`, `MESSMERS_FLAME_AEGIS`, `SUNS_PROTECTION`, `MESSMERS_VENOM`, `ASGARDIANS_FLIGHT`, `SHURIMAN_ENDURANCE`) activate within 1 tick of swap-in instead of up to 80; swap-out is equally instant via the same refresh call's diff pass.

Three independent mechanisms live in this file:

0. **Durability regen block** — a separate `% 243` cadence check runs *before* the `% 80` early return, so it ticks independently of the bonus-refresh path. For every online player × armor slot, if the equipped stack carries `REGENS_DURABILITY` and has non-zero damage, decrement its damage by 1. Used by the Knight Radiant Shard set to give it elytra-style self-repair.

1. **`BONUSES` dispatch loop** — a `List<Bonus>` where each `Bonus` carries a `Predicate<TickContext>` trigger and one-or-more `Effect`s (status effects), re-applied every tick while the trigger matches. `Bonus.effects(...)` is the only factory. `TickContext` bundles all six equipment slots + `isNight` + dimension, with helpers `hasFullSet(setId)` and `inTimelessDimension()` (Nether/End treated as always-dark for time-gated bonuses). Current entries:
   - `night_vision_helmet` — helmet has `GRANTS_NIGHT_VISION` → Night Vision I.
   - `mistborn_set_night` — all four armor slots share `SET_ID="mistborn"` **and** it's night or Nether/End → Strength I + Speed I. The hidden, time-gated Mistborn payoff.
   - `pacifist_full_set` — all four armor slots share `SET_ID="pacifist"` (no time gate) → Regeneration I.
   - `sages_grace_mainhand` — main-hand stack has `HEALS_TARGET` → Sage's Grace (cosmetic HUD badge; the actual heal-on-hit lives in `AttackHandlers`' `AttackEntityCallback` branch, gated on the same mainhand flag). No time gate. Applied with `showParticles=false` so the vanilla ambient bubbles don't radiate.
   - `water_breathing_helmet` — helmet has `GRANTS_WATER_BREATHING` → Water Breathing I.

## `item/AttackHandlers.java`

Registers two separate hooks because Fabric's two event channels cover different damage paths:

1. **`ServerLivingEntityEvents.ALLOW_DAMAGE`** (pre-damage, fires for *every* damage path since they all funnel through `LivingEntity.damage`) — two additive branches. (a) If the attacker is a player with `MESSMERS_VENOM` and their weapon stack is in `#minecraft:spears`, apply Messmer's Venom (14s, amp 0) to the victim. Gating on the weapon tag (not the set) means melee stabs, lunge dashes, and thrown spear hits all propagate, while fists and non-spear weapons don't — any future modded spear tagged `#minecraft:spears` inherits the behavior for free. (b) If the attacker is a player whose **main-hand** stack carries `APPLIES_WITHER_ON_HIT`, apply Wither II for 4s to the victim. Checked against `player.getMainHandStack()` rather than `source.getWeaponStack()` so off-hand flagged stacks don't propagate — matches the axe's `AttributeModifierSlot.MAINHAND`-gated attack-speed modifier. Always returns `true` — both branches are additive propagation, not cancellations.

2. **`AttackEntityCallback.EVENT`** (pre-damage, left-click melee only — fires inside `PlayerEntity.attack` and misses lunge dashes + projectile hits) — if the attacker's main-hand stack has `HEALS_TARGET`, replicate vanilla's private `isCriticalHit` check + cooldown gate, heal the target (if `LivingEntity`) by 4 HP on a normal hit or 8 HP on a crit, spawn heart particles, play the vanilla attack-sound (STRONG/WEAK/CRIT), consume 3 durability via `stack.damage(...)` (since vanilla's wear path is short-circuited by our cancel), reset the attack cooldown, and return `ActionResult.SUCCESS` to cancel vanilla damage/knockback/sweep/durability. Side effects (heal, sounds, particles, durability) are gated to `ServerWorld` to avoid desync and double-broadcast; the cooldown reset + cancel return run on both sides. Sweep is implicitly defeated — `AttackEntityCallback` never fires for sweep-caught entities, and `SUCCESS` short-circuits `PlayerEntity.attack` before `doSweepingAttack` runs.

Pattern choice: use **`AttackEntityCallback`** for behavior that must *replace* the damage/knockback (heal, freeze, teleport, etc.); use **`ALLOW_DAMAGE`** for behavior that runs *additively* alongside normal damage and needs to cover non-melee paths too (debuffs, bleed, lifesteal).

## `item/BlockBreakHandler.java`

Bonus-diamond drops for `BONUS_DIAMOND_CHANCE`-flagged tools (currently just Just Hit Bro, the Esh-Endra-Navesh pickaxe). Registers one hook on `PlayerBlockBreakEvents.AFTER`, which fires post-break — the block has already been consumed and its own drops have spawned, so the handler layers phantom diamond drops on top rather than replacing anything.

Guards: `ServerWorld` only (AFTER fires client-side too for prediction), server-side `ServerPlayerEntity` only, creative/spectator excluded. Then reads `BONUS_DIAMOND_CHANCE` off the player's **main-hand** stack — so the pickaxe must be the tool that actually broke the block, not just present in the inventory.

Roll: `serverWorld.random.nextInt(800) != 0` early-returns — 1/800 per break. On success, `Block.dropStacks(Blocks.DIAMOND_ORE.getDefaultState(), world, pos, blockEntity=null, player, tool)` runs vanilla's `blocks/diamond_ore` loot table verbatim with the same tool the player used, so **Fortune rolls the raw-diamond multiplier and Silk Touch drops a diamond_ore block** exactly as they would against a real ore break. Then plays `ModSounds.BONUS_DIAMOND` at the block position on `SoundCategory.PLAYERS` (rides the Players volume slider — this is gear-event, not a world-ambient sound), `source=null` so nearby clients also hear the proc.

The roll probability is the single tuning knob, stored as `BONUS_DIAMOND_ROLL_DENOMINATOR` at top of file. Adjust there, not at each call site.

Bounty Hunter — the leggings-gated arrow-kill bonus drop for `BOUNTY_HUNTER`-flagged Raider's Legacy legs (Trousers of the Trail). Event-driven via `ServerLivingEntityEvents.AFTER_DEATH`, registered from `onInitialize`. Server-side only.

## `item/MjolnirMaceItem.java`

`MaceItem` subclass backing Mjolnir, the Thor signature weapon (and 5th piece of the 5/5 Thor set). Right-click on the main hand starts a `UseAction.SPEAR` charge for `CHARGE_TICKS` (10), then `onStoppedUsing` launches the player along their look vector with vanilla `TridentItem`'s Riptide math. Launch magnitude is looked up dynamically via `EnchantmentHelper.getTridentSpinAttackStrength(stack, player)` at `MjolnirMaceItem.java:74` — the same call vanilla `TridentItem` uses — so the launch **scales with the Riptide enchant level on the stack**; this is **not cosmetic/visual flavor** as earlier docs claimed. Mjolnir ships with **Riptide III** (the post-datapack-override max from `data/minecraft/enchantment/riptide.json`'s `max_level: 3`); see the datapack tag overrides section below for how Mjolnir is opted into Riptide via the custom `famspecial:riptide_eligible` tag. No water, no rain, no Riptide enchant required to trigger (level 0 still launches — the Riptide level only tunes magnitude). The launch is gated three ways: hand must be `MAIN_HAND`, the stack must carry `THOR_MACE`, and the player must currently have `ASGARDIANS_FLIGHT` — any failure short-circuits to `super.use` / `super.onStoppedUsing` and the mace behaves like a vanilla mace.

**Asgardian's Flight as gameplay predicate.** `MjolnirMaceItem` reads `player.hasStatusEffect(ASGARDIANS_FLIGHT)` rather than re-deriving the 5/5 + mainhand gate at the call site. The effect is the source of truth — applied by `ArmorEffects`' `asgardians_flight_full_thor_set_and_mjolnir` bonus and instant-stripped via `ArmorEffects.refreshBonusesFor(player)` called from the equip / slot / click-slot mixins when either half drops. Decoupling the gate this way means a mid-charge armor/hotbar/inventory swap drops the launch on the next tick without `MjolnirMaceItem` having to re-walk the equipment slots.

**Client-authoritative velocity.** `onStoppedUsing` mirrors vanilla `TridentItem` byte-for-byte — no `if (world.isClient()) return true;` early exit. The launch math must run on **both** client and server: `LocalPlayerEntity` is client-authoritative on its own velocity, so a server-only `addVelocity` becomes an `EntityVelocityUpdateS2CPacket` that the local player ignores, producing spin-without-launch (the v1.2.5 bug). Do not re-introduce the client-side early return. Only the cooldown arming (`getItemCooldownManager().set(stack, RIPTIDE_SPIN_TICKS)`) is server-only — the client receives that via the cooldown packet on the next tick.

**Per-launch durability cost.** Every successful launch — whether it's the vanilla water/rain riptide path or Asgardian's Flight's dry-land launch — costs **−1 durability** on Mjolnir. Applied server-side at `MjolnirMaceItem.java:91` via `stack.damage(1, player, EquipmentSlot.MAINHAND)`, after the cooldown is set and immediately before `onStoppedUsing` returns. Gated on `!world.isClient()` so the wear is authoritative from the server and not double-counted on the client. Too-short charges (below `CHARGE_TICKS`) are early-returned via `super.onStoppedUsing` at `MjolnirMaceItem.java:61` and **do not** incur the cost — durability is only consumed when the launch actually fires.

## `item/FortuneGloryItem.java`

`CrossbowItem` subclass backing **Fortune & Glory**, the Raider's Legacy signature crossbow. Sibling to `MjolnirMaceItem` in the signature-weapon-subclass discipline: **must be registered in `RaidersLegacyItems.register()` from `FamSpecial.onInitialize` before `ModItemGroups.registerItemGroups()` runs** — registering an `Item` subclass at static-final class-load crashes in 1.21+ because `SimpleRegistry` rejects intrusive holders during the early-boot window the creative-tab-entries lambda runs in.

**Three-shot scheduler — Crusader's Volley.** A single right-click charge fires **three shots back-to-back** at a snappy ~2-tick cadence (~100 ms gap between each, ~200 ms total spread), with all three shots matching the originally-loaded projectile type (arrows OR firework rockets). The `use` override calls `super.use(...)` first so vanilla fires shot 1 exactly as it would for any crossbow, then — gated on the stack carrying `CRUSADERS_VOLLEY` and the user being a `ServerPlayerEntity` — appends **two pending shots** into a private `ConcurrentLinkedQueue<ScheduledShot>` (no per-player keying — `ScheduledShot` carries the player UUID directly). The loaded projectile is captured from the stack's `CHARGED_PROJECTILES` component **before** `super.use(...)` clears it and stashed on the `ScheduledShot` record so shots 2 and 3 can replay the right entity type even though the component is gone by the time they fire. Constants: `SHOT_2_DELAY_TICKS = 2`, `SHOT_3_DELAY_TICKS = 4`, so shot 1 at t=0, shot 2 at t=2, shot 3 at t=4. A `ServerTickEvents.END_SERVER_TICK` handler (registered from `FortuneGloryItem.registerTickHandler()`, wired from `FamSpecial.onInitialize`) drains the queue every server tick: any entry whose target tick has elapsed fires via `fireScheduledShot`, which branches on the captured projectile — `FireworkRocketItem` → spawn a `FireworkRocketEntity` matching vanilla `CrossbowItem`'s firework path; otherwise → spawn a `VolleyArrowEntity` (the iframe-bypass arrow path described below).

**Iframe bypass — VolleyArrowEntity.** The 2-tick cadence is below vanilla's `LivingEntity.timeUntilRegen` 10-tick invulnerability window, so shots 2 and 3 would normally be silently rejected as equal-or-lesser-damage during iframes. Shots 2 and 3 spawn a `VolleyArrowEntity` (in `entity/VolleyArrowEntity.java`, registered via `entity/ModEntities.java`) instead of a vanilla `ArrowEntity`. `VolleyArrowEntity` overrides `onEntityHit(EntityHitResult)` and zeroes the target `LivingEntity.timeUntilRegen` BEFORE delegating to `super.onEntityHit(...)`, so vanilla's iframe gate sees no cooldown and applies damage normally. Shot 1 stays vanilla — it's the first hit and doesn't need bypass. Everything else about hit behavior (Multishot/Piercing inheritance, damage scaling, knockback) stays vanilla because `VolleyArrowEntity` only touches the iframe field.

**Multishot/Piercing inheritance.** Shots 2 and 3 read the source crossbow stack's enchantment levels at fire time (from the stashed stack snapshot) and inherit Multishot and Piercing exactly as vanilla would for a fresh shot — so **Multishot III + Crusader's Volley = 9 arrows per release**. This is intentional per the user's design — the volley-style mechanic stacks multiplicatively with the crossbow's own enchant pool, not additively. Note: a companion enchant-block carve-out makes vanilla means of applying Multishot to Fortune & Glory unavailable; the inheritance code is defensive, kept in for creative `/enchant` and NBT-tool paths.

**Ammo / durability discipline.** "Single reload, triple shot." Shots 2 and 3 do **not** consume additional projectiles from the player's inventory and do **not** wear the crossbow's durability beyond what vanilla took for shot 1 — the queue-drain just re-fires the cached arrow path against the world without re-running vanilla's ammo-consume / wear-tick code. `Stats.USED.getOrCreateStat(Items.CROSSBOW)` is incremented once per shot (3 increments per release) so the player's stats screen reflects the true shot count. If the user disconnects or dies before the cadence completes, pending entries for that UUID are removed via the disconnect / `AFTER_DEATH` listeners.

The HUD-badge advertisement lives in `ArmorEffects`' `crusaders_volley_mainhand` bonus on the same `CRUSADERS_VOLLEY` flag so badge + mechanic share one source of truth.

## Datapack tag overrides (Mjolnir enchant eligibility)

The mod ships a small set of resource-pack JSONs under `src/main/resources/data/` that opt **Mjolnir** into a curated subset of vanilla enchantment pools. This is a **runtime-data mechanism, not a mixin** — no code changes are involved; the JSONs ship with the jar and Mojang's data loader merges them with the vanilla tags at load time.

**Why tag overrides instead of a mixin.** Vanilla `Enchantment#isAcceptableItem(stack)` checks the enchantment's `supported_items` tag against the **registered item id**. Mjolnir is registered as `famspecial:mjolnir`, not `minecraft:mace`, so it falls outside any vanilla tag that lists items by id (`minecraft:enchantable/mace`, `minecraft:enchantable/weapon`, etc.). Tag overrides are the cleanest way to opt a custom-id item into specific vanilla enchant pools without touching vanilla code or layering a mixin on `isAcceptableItem`.

**The vanilla `enchantable/*` tag overrides** — all under `src/main/resources/data/minecraft/tags/item/enchantable/`, all using `{"replace": false, "values": [...]}` so they're additive to vanilla:

- `durability.json` — opts **`famspecial:mjolnir`** and **`famspecial:fortune_and_glory`** into **Mending / Unbreaking / Curse of Vanishing**.
- `mace.json` — opts Mjolnir into **Density / Breach / Wind Burst** (the mace-specific pool).
- `weapon.json` — opts Mjolnir into **Smite / Bane of Arthropods**. Sharpness is intentionally **not** enabled — it gates on `minecraft:enchantable/sharp_weapon`, which Mjolnir is deliberately left out of.
- `fire_aspect.json` — opts Mjolnir into **Fire Aspect**.
- `crossbow.json` — opts **`famspecial:fortune_and_glory`** into **Quick Charge / Multishot / Piercing** (the crossbow-specific pool). Same additive-override mechanism as the Mjolnir entries above. **Multishot is then gated back out** by the targeted Multishot override below — Fortune & Glory still appears in this tag, but the Multishot enchantment no longer reads from this tag (see "The Multishot-exclusion carve-out" below).

**The Riptide-only carve-out** uses two files (already shipped before this change):

- `src/main/resources/data/famspecial/tags/item/riptide_eligible.json` — a custom item tag whose value is `["#minecraft:enchantable/trident", "famspecial:mjolnir"]`, i.e. the union of vanilla trident-enchantables and Mjolnir.
- `src/main/resources/data/minecraft/enchantment/riptide.json` — a vanilla-enchantment **override** that copies the rest of vanilla Riptide verbatim (anvil_cost, effects, exclusive_set, max_level, etc.) but swaps `supported_items` from `#minecraft:enchantable/trident` to `#famspecial:riptide_eligible`. Adding Mjolnir straight to `enchantable/trident` would also opt it into Loyalty + Channeling, which we explicitly don't want — overriding Riptide alone is the only enchant we want to extend.

**The Multishot-exclusion carve-out** (Fortune & Glory) is the mirror image of the Mjolnir/Riptide pattern — instead of opting a custom item *into* a vanilla enchant, it gates a vanilla item *out of* one. **Why:** the previous "6-arrow chaos" intent for Fortune & Glory is being walked back; Multishot is no longer wanted on the crossbow. **What:**

- `src/main/resources/data/famspecial/tags/item/multishot_eligible.json` — a custom item tag whose value is `["minecraft:crossbow"]` only. Fortune & Glory is intentionally **absent**.
- `src/main/resources/data/minecraft/enchantment/multishot.json` — a vanilla-enchantment **override** that copies vanilla Multishot byte-for-byte (anvil_cost, effects, exclusive_set, max_cost, max_level, min_cost, slots, weight, description) but swaps `supported_items` from `#minecraft:enchantable/crossbow` to `#famspecial:multishot_eligible`. Because the new tag omits `famspecial:fortune_and_glory`, Multishot's `isAcceptableItem` check now returns false for the custom crossbow.

**The arrangement.** All other crossbow enchants (Quick Charge, Piercing) continue to read from the unmodified `#minecraft:enchantable/crossbow` tag, which Fortune & Glory still belongs to via the additive `crossbow.json` override above; durability enchants (Mending, Unbreaking, Curse of Vanishing) still come from the additive `durability.json` override. Only Multishot now reads from the more restrictive `#famspecial:multishot_eligible` tag. Vanilla `minecraft:crossbow` is unaffected — it's listed in the new tag, so it still receives Multishot normally. The parallel multishot-inheritance code in `FortuneGloryItem.fireScheduledShot` (the second-shot fire path) is left in as defensive — fires correctly if a creative-mode `/enchant` command bypasses the data layer.

**Net effect: Fortune & Glory = all vanilla crossbow enchants except Multishot.** Accepts Quick Charge, Piercing, Mending, Unbreaking, Curse of Vanishing. **Does not** accept Multishot.

**Net effect: Mjolnir = vanilla mace enchants + Riptide only.** Accepts Density / Breach / Wind Burst, Smite / Bane, Fire Aspect, Mending / Unbreaking / Curse of Vanishing, and Riptide. **Does not** accept Sharpness, Loyalty, or Channeling.

**When to reach for this technique.** Any time a custom-id item should accept a vanilla enchant the vanilla tag doesn't list it under. To opt into an entire vanilla enchant pool, drop a `{"replace": false, "values": [...]}` JSON at `data/minecraft/tags/item/enchantable/<pool>.json`. To accept an individual enchant whose pool you don't want to fully extend, define a custom item tag under `data/<modid>/tags/item/<name>.json` and override the vanilla enchantment JSON at `data/minecraft/enchantment/<enchant>.json` with `supported_items` pointing at your custom tag. No mixin, no code wiring — rebuild and ship.

## Custom-id item parity for vanilla advancements/stats

Same root cause as the enchant-tag overrides above — `famspecial:mjolnir` and `famspecial:fortune_and_glory` aren't `minecraft:mace` / `minecraft:crossbow` by id, so vanilla advancement criteria that pin on those item ids (the four crossbow advancements `ol_betsy` / `arbalistic` / `two_birds_one_arrow` / `whos_the_pillager_now`, plus the mace advancement `overoverkill`) won't trigger when the custom-id items are used, and vanilla per-item stats slots (`Stats.USED.getOrCreateStat(Items.CROSSBOW/MACE)`) won't increment either. Fixed in two coordinated places: (1) two custom item tags `data/famspecial/tags/item/counts_as_crossbow.json` and `counts_as_mace.json` define the union of vanilla id + custom id, then five vanilla advancement JSONs are overridden under `data/minecraft/advancement/adventure/` byte-for-byte except for the single `items` field swapped to `#famspecial:counts_as_crossbow` or `#famspecial:counts_as_mace`; (2) `MjolnirMaceItem#onStoppedUsing` and both `FortuneGloryItem#use` (first shot path) and `FortuneGloryItem#fireSecondShot` manually fire `Stats.USED.getOrCreateStat(Items.MACE/CROSSBOW)` on top of the vanilla-counterpart stat slot the `MaceItem` / `CrossbowItem` superclasses already increment for the custom id, so the player's stats screen shows non-zero entries for **both** the custom and vanilla counterparts. **Future custom-id items that should satisfy a vanilla advancement / increment a vanilla stat should follow the same `counts_as_<vanilla_id>` tag convention plus a manual `Stats.USED` increment in the use/release path.**

## Chestplate→elytra anvil merge (attribute-modifier trade-off)

Cross-references the merge implementation in `AnvilScreenHandlerMixin` (`MIXINS.md`). Two SYSTEMS-relevant side effects:

1. **Armor points traded for flight.** `famspecial$buildMergedElytra` filters the source chestplate's `ATTRIBUTE_MODIFIERS` component before copying it onto the elytra output: entries whose attribute is `EntityAttributes.ARMOR` or `EntityAttributes.ARMOR_TOUGHNESS` are skipped, but every other modifier (MAX_HEALTH, MOVEMENT_SPEED, KNOCKBACK_RESISTANCE, any future custom attribute) is copied through. Net effect: the merged elytra trades the chestplate's diamond-baseline armor points for flight while retaining the per-theme stat bonuses.
2. **Custom-component transplant.** Mod components are carried via `ModComponents.copyModComponentsForElytra(source, out)` — see the `ModComponents` section for the `MOD_COMPONENTS_BASE` / `ELYTRA_EXCLUDED_COMPONENTS` split.

Identity-gating and the Mending/Unbreaking enchant merge (both inputs contribute; max level wins on duplicates) live in the mixin itself; see `MIXINS.md` for the merge eligibility predicate (`MERGE_ELIGIBLE_SETS` + `IS_FAMSPECIAL_GEAR` forgery gate) and the enchant filter.

**Thor armor MAX_HEALTH cross-reference.** The 4 Thor armor pieces (Thunderhelm, Asgardian Plate, Warrior's Greaves, Stormstrider) each carry **+1 MAX_HEALTH** via `ATTRIBUTE_MODIFIERS` components built in `ThorItems.buildThorArmor` (`ThorItems.java:117` onward; per-piece constants `HELM_MODIFIERS` / `PLATE_MODIFIERS` / `GREAVES_MODIFIERS` / `STRIDER_MODIFIERS` at lines 112–115). A full 4-piece Thor armor set therefore adds **+2 hearts**. The bonus carries through the chestplate→elytra merge because `MAX_HEALTH` is not in the attribute-filter skip list above — so a Thor-chestplate-merged elytra keeps its +1 HP contribution even though the diamond armor/toughness baseline is stripped. Primary per-piece documentation lives in `ITEMS.md`; this cross-reference exists so SYSTEMS readers tracing the merge's attribute behavior can trace the Thor bonus to its source.

## `item/ThorEffects.java`

Event-driven Thor handlers — Mjolnir's lightning-on-hit roll plus Warrior's Greaves' Storm's Awakening kill trigger. Two listeners registered from `onInitialize`:

2. **`ServerLivingEntityEvents.AFTER_DEATH`** — Storm's Awakening. When the killer is a player wearing Warrior's Greaves (leggings flagged `TRIGGERS_STORM_AWAKENING`) and the per-player cooldown is clear, roll the trigger chance — **`STORM_TRIGGER_CHANCE = 0.08f`** (flat 8% per kill, at `ThorEffects.java:54`; this is the production value, down from the `1.00f` temporarily used during testing) — to start a thunderstorm. **If `world.isThundering()` is already true, skip the kill entirely — no roll, no cooldown burn.** On a successful trigger, play `SoundEvents.ENTITY_LIGHTNING_BOLT_THUNDER` on `SoundCategory.WEATHER` at the killer's position, **vol 10000, pitch 0.8**, immediately before `world.setWeather(0, STORM_THUNDER_DURATION_TICKS, true, true)` — **`STORM_THUNDER_DURATION_TICKS = 2400`** (~2 min real-time, at `ThorEffects.java:60`; down from the earlier 4800-tick / ~4 min value). The vol-10000 hack abuses vanilla's volume-to-attenuation-range scaling so the thunderclap reaches every player in the world (not just nearby) — this is the audible "the storm answered" cue. ## `client/HealthOverlay.java`

Client-only world-space HP bar overlay above remote players. Registered on `WorldRenderEvents.AFTER_ENTITIES`; runs every frame but bails immediately if the local player's HEAD slot isn't flagged `SHOWS_ENTITY_HP` (Sage's Crown). When active, iterates `world.getPlayers()` (narrower/cheaper than full entity iteration), skips self + removed/dead/spectator players + anything beyond `VIEW_DISTANCE_SQ` (48²), and draws a two-quad bar (dark bg + HP-colored fg) above each remaining player's head. Billboards via `matrices.multiply(camera.getRotation())`, scale `0.025` (matches vanilla nameplate scale), head clearance `0.55` blocks above the entity. Uses `RenderLayers.debugQuads()` (POSITION_COLOR pipeline) — text-background layers silently drop writes due to a vertex-format mismatch. Three-band color ramp: green ≥ 66%, yellow ≥ 33%, red otherwise. Purely client-side — health is read live off the tracked client entity, no server involvement.
