# MIXINS.md

SpongePowered Mixin entries in `src/main/java/github/rahularora375/famspecial/mixin/` plus the `famspecial.mixins.json` manifest. Root context: `CLAUDE.md`. For the components/effects/systems the mixins interact with, see `SYSTEMS.md`; for item-side flag origins, see `ITEMS.md`.

## `mixin/` + `famspecial.mixins.json`

**SpongePowered Mixins** wired in via `fabric.mod.json` → `"mixins": ["famspecial.mixins.json"]`. Responsibilities: hook into vanilla item behavior for set bonuses, layer custom damage scaling, suppress targeted enchant side effects, enforce elytra-style durability caps, widen access to protected fields for the Necromancer summon's target-goal swap, drive the instant-apply / instant-strip dispatch on every equipment-state change (armor equip, hotbar swap, survival inventory click, creative inventory drag) via a shared `ArmorEffects.refreshBonusesFor(player)` call, and power the themed-chestplate → vanilla-elytra anvil merge feature (plus its grindstone re-stamp). **Themed gear behaves identically to vanilla gear for anvil and grindstone operations in every case except the chestplate→elytra merge** — the mod no longer gates rename, repair-combine, enchant-merge, or disenchant on any themed stack. This applies in both creative and survival; any creative-vs-survival asymmetries the player observes (e.g. Thorns III on an elytra in creative) are vanilla Mojang behavior, not mod behavior.

- `GrindstoneScreenHandlerMixin` — `@Inject` at `TAIL` of `updateResult()`. **Single-purpose: merged-elytra identity re-stamp.** All other grindstone behavior is vanilla — themed gear (weapons, armor, non-merged items) is grindstone-able exactly like vanilla gear, in both creative and survival. The re-stamp guard (`GrindstoneScreenHandlerMixin.java:30`): if `result` is non-empty and either input is a themed elytra (`SET_ID` + `Items.ELYTRA`), call `ModComponents.copyModComponentsForElytra(inputN, result)` — preferring `input0` over `input1` when both qualify, so the re-stamp is deterministic. Rationale: vanilla grindstone drops the custom component bundle when stripping enchants off a themed elytra; this step restores `SET_ID`, `IS_FAMSPECIAL_GEAR`, `REGENS_DURABILITY`, `BLOCKS_MENDING`, and every other custom flag except `INDESTRUCTIBLE` (excluded by design in `copyModComponentsForElytra`) so the disenchanted elytra keeps its themed identity and set-bonus membership. The `!result.isEmpty()` guard skips the re-stamp when the two inputs aren't a valid grindstone pair (vanilla already blanks the output in that case). **Removed in the recent refactor:** the `HEALS_TARGET` and `SET_ID + !elytra` blank-output branches — themed weapons and themed non-elytra armor are no longer gated.
- `AnvilScreenHandlerMixin` — one inject on `AnvilScreenHandler.updateResult()` (fields `input` / `output` are inherited from `ForgingScreenHandler`, which the mixin extends because Mixin can't `@Shadow` inherited fields in dev mappings — the synthetic constructor is stripped at apply time). **Single-purpose: themed-chestplate → vanilla-elytra merge.** All other anvil behavior is vanilla — rename, repair-combine, enchant-merge, and level-cost math on themed gear all behave exactly as they do on vanilla gear, in both creative and survival. The mod no longer gates Mjolnir anvil input, `HEALS_TARGET` items, or `BLOCKS_MENDING`+Mending combinations; those TAIL blank-output guards have been removed entirely.

  **HEAD inject `famspecial$mergeThemedChestplateElytra`** (`AnvilScreenHandlerMixin.java:89`, `cancellable = true`) — the merge entry point. Predicate, evaluated in order (all must hold, any failure falls through to vanilla `updateResult`):
  1. Left input's item is `Items.DIAMOND_CHESTPLATE`.
  2. Left input's `SET_ID` is in the 10-theme `MERGE_ELIGIBLE_SETS` allowlist: `mistborn`, `pacifist`, `poseidon`, `fire_serpent`, `necromancer`, `knight_radiant`, `esh_endra_navesh`, `shurima`, `thor`, `raider`. OldFam is intentionally excluded (its pieces don't carry `SET_ID`); future themes opt in explicitly by name. The Raider chestplate (The Explorer's Jacket) carries +1 MAX_HEALTH (+0.5 hearts) on the chest slot, which carries through the merge automatically via `copyModComponentsForElytra`'s attribute filter (which strips only ARMOR / ARMOR_TOUGHNESS).
  3. Left input carries `IS_FAMSPECIAL_GEAR=true`. This is the NBT-forgery gate — the flag is stamped exclusively by `ModItemGroups`, so a plain diamond chestplate with a hand-stamped `SET_ID` is rejected.
  4. Right input is `Items.ELYTRA`.
  5. Right input carries no `SET_ID` (prevents re-merging an already-themed elytra).

  On match, `famspecial$buildMergedElytra(chestplate, elytra)` assembles the output stack: a fresh `new ItemStack(Items.ELYTRA)` carrying (a) the source chestplate's `LORE`; (b) the source chestplate's `ATTRIBUTE_MODIFIERS`, rebuilt via `AttributeModifiersComponent.Builder` with any entry whose attribute equals `EntityAttributes.ARMOR` or `EntityAttributes.ARMOR_TOUGHNESS` filtered out (the diamond baseline intentionally does not leak onto the elytra slot — this is the armor-for-flight trade); (c) a custom name pulled from the shadowed `newItemName` rename field when non-blank, otherwise the chestplate's `CUSTOM_NAME`; (d) **enchants combined from BOTH inputs** — a fresh `ItemEnchantmentsComponent.Builder` seeded from `ItemEnchantmentsComponent.DEFAULT`, fed both `chestplate.getEnchantments()` and `elytra.getEnchantments()` through `mergeAllowedEnchants(builder, source)` which filters to entries whose registry key `matchesKey(Enchantments.MENDING)` or `matchesKey(Enchantments.UNBREAKING)` only. `ItemEnchantmentsComponent.Builder#add` keeps the **max level** when the same enchant is contributed by both chestplate and elytra, so feeding a Mending chestplate + a Mending-II elytra yields Mending II on the output. All other enchants (Protection, Unbreaking-III-like surrounding levels aside of the cap, Thorns, etc.) are stripped. Result written via `EnchantmentHelper.set(out, builder.build())`; (e) every custom mod component via `ModComponents.copyModComponentsForElytra(chestplate, out)`, which walks `MOD_COMPONENTS_BASE` but skips `ELYTRA_EXCLUDED_COMPONENTS` — `INDESTRUCTIBLE` is excluded by design so the merged elytra remains breakable, preventing an infinite-flight exploit. `SET_ID`, `IS_FAMSPECIAL_GEAR`, `REGENS_DURABILITY`, `BLOCKS_MENDING`, and all other themed flags do carry through, so the worn elytra still completes the set and drives its `ArmorEffects` bonuses.

  Level cost is set to a flat `4` on the shadowed `levelCost` `Property`. The inject logs an INFO line (`"Merged {} chestplate into elytra; result SET_ID={}"`) via `FamSpecial.LOGGER` on success, then calls `sendContentUpdates()` explicitly and `ci.cancel()`s vanilla's body. HEAD (not TAIL) is required because vanilla `updateResult` early-returns mid-method for incompatible item pairs — chestplate + elytra aren't same-item, and elytra isn't a repair material — and `@At("TAIL")` only fires at the last RETURN instruction, so an early return bypasses TAIL. Non-matching inputs fall through to vanilla `updateResult`.
- `TridentItemMixin` — two `@Redirect`s on `TridentItem`'s `use` and `onStoppedUsing` methods, each targeting the `PlayerEntity.isTouchingWaterOrRain()Z` INVOKE site that vanilla uses to gate riptide. If the player is actually touching water/rain, returns `true` (vanilla behavior). Otherwise, if the trident is in the **main hand** and the player wears a full `SET_ID="poseidon"` armor set (checked via `ArmorEffects.hasFullSet(player, "poseidon")`), returns `true` anyway — riptide fires on dry land. Off-hand use falls back to vanilla. The INVOKE target owner must be `PlayerEntity` (not `Entity`) because javac bakes the compile-time static receiver type into the INVOKEVIRTUAL instruction, and both vanilla call sites use a `PlayerEntity`-typed reference. The `use` redirect grabs `Hand` via MixinExtras' `@Local(argsOnly = true)`; the `onStoppedUsing` redirect reads it off `self.getActiveHand()`.
- `LivingEntityMixin` — five MixinExtras `@ModifyReturnValue` handlers, all on `LivingEntity.modifyAppliedDamage` at `RETURN`. (1) Scales incoming fire damage to 20% when the victim has `MESSMERS_FLAME_AEGIS` (acts like Fire Protection, not Fire Resistance; stacks multiplicatively with vanilla armor/protection math since it runs at the end of that same method). Vanilla Fire Resistance short-circuits fire damage to zero earlier in `LivingEntity.damage` before this method is called, so having both at once still results in zero damage. (2) When the attacker is a `LivingEntity` holding a stack flagged `GRANTS_SHARDBEARING` in their main hand, add 7% of the target's current HP as bonus damage to the final post-mitigation total (factor `0.07f` at `LivingEntityMixin.java:61`, up from the previous 5%). Hooked on `modifyAppliedDamage` rather than `applyArmorToDamage` so the chip lands AFTER all mitigation — unlike a pre-armor bonus, it always contributes on unarmored mobs (the common case). `self.getHealth()` at this point reflects HP BEFORE the current hit lands. Non-attacker damage sources and non-`LivingEntity` attackers short-circuit. Projectile / indirect hits only chip if the attacker is STILL holding the sword at impact time. (3) When the victim's chest slot carries `GRANTS_UNDEAD_RESISTANCE` AND the attacker's type is in `EntityTypeTags.UNDEAD`, multiply damage by 0.4 (60% reduction). `source.getAttacker()` returns the shooter for arrow/projectile sources, so skeleton arrows + wither-skeleton melee both fold into one branch — no separate projectile path. Same multiplicative placement as (1), so it stacks correctly with armor/protection. (4) When the victim has `SUNS_PROTECTION`, multiply damage by 0.8 (Resistance-I-equivalent 20% reduction). Effect-presence is the gate, not a component lookup — `ArmorEffects` applies the effect only while the boots are worn AND the player is in a desert-family biome (`BiomeKeys.DESERT`, `BADLANDS`, or `ERODED_BADLANDS`). `SUNS_PROTECTION` is **not** in `MOD_MANAGED` — on boots removal or biome leave the trigger simply stops re-upping and the effect decays naturally over ~20 s (the applied duration). Same multiplicative placement as the other three. (5) `famspecial$technobladeFallImmunity` — when the victim is a player with `TECHNOBLADE_NEVER_DIES` AND `source.isOf(DamageTypes.FALL)`, return `0.0f`. Raider's Legacy 4/4 fall-damage immunity. Effect-presence is the gate (applied only by `ArmorEffects`' `technoblade_never_dies_full_set` bonus while all four armor slots share `SET_ID="raider"`); in `MOD_MANAGED`, so breaking the set instant-strips the effect within 1 tick via the equip-mixin `refreshBonusesFor` dispatch.
- `LivingEntityKnockbackMixin` — MixinExtras `@ModifyExpressionValue` on `LivingEntity.takeKnockback(DDD)V`, targeting the `INVOKE`-site of `LivingEntity#getAttributeValue(RegistryEntry)D`. The only `getAttributeValue` call in `takeKnockback` reads `KNOCKBACK_RESISTANCE` (verified against yarn 1.21.11+build.4); when the receiver's attacker is a `LivingEntity` carrying `EMPERORS_DIVIDE`, return `0.0` from the lookup. Vanilla then computes `(1 - 0.0) = 1`, so knockback lands at full strength regardless of the victim's KB-resistance attribute. Backs the **Sun Disc Spear** (Shurima) gameplay — the effect itself is applied by `ArmorEffects`' `emperors_divide_mainhand` bonus while the spear is in the attacker's mainhand.
- `PlayerEntityExhaustionMixin` — `@Inject(at = HEAD, cancellable = true)` on `PlayerEntity.addExhaustion(F)V`. Cancels the call when `self.hasStatusEffect(SHURIMAN_ENDURANCE)` returns true. Every saturation-drain path (sprint, jump, attack, regen-from-food, block break, mob damage) funnels through `addExhaustion`, so a single HEAD-cancel covers all of them — no per-source instrumentation. Backs the **Shuriman Endurance** Shurima 4/4 set bonus; the effect is applied by `ArmorEffects` with a 4-minute apply-once duration so once earned in the desert the saturation lock persists anywhere the player goes for the full ~4 minutes.
- `ApplyExhaustionEnchantmentEffectMixin` — `@Inject(at = HEAD, cancellable = true)` on `ApplyExhaustionEnchantmentEffect.apply(ServerWorld, int, EnchantmentEffectContext, Entity, Vec3d)`. Cancels when `context.stack()` carries the `NO_LUNGE_HUNGER` flag — short-circuits Lunge's post-piercing-attack exhaustion for Fire Serpent's Wrath. Other lunge effects (damage, knockback) still apply; only the exhaustion tick is suppressed. Generalizes to any future enchant that reuses `ApplyExhaustionEnchantmentEffect`.
- `ItemStackMixin` — `@Inject(at = HEAD, cancellable = true)` on the core `ItemStack.damage(int, ServerWorld, ServerPlayerEntity, Consumer<Item>)` overload. Every other `damage(...)` path funnels through this one (armor wear's `(int, LivingEntity, EquipmentSlot)` delegates here, weapon wear's `(int, LivingEntity, Hand)` forwards to the slot overload, dispenser wear calls this directly), so a single inject covers all damage paths. When the stack carries `INDESTRUCTIBLE`, clamps damage so `getDamage()` never exceeds `getMaxDamage() - 1` — if the amount would overflow, it's clamped via `setDamage(...)` and the break callback is skipped. Mirrors vanilla elytra's `willBreakNextUse()` gate: the item sticks at 1 durability forever. On a fresh not-broken → broken transition, the mixin also plays `ENTITY_ITEM_BREAK` once at the player's position (null source so the wearer hears it) and walks the 4 armor slots to find the owning one, then calls `player.equipStack(slot, stack)` — that routes through `LivingEntity.onEquipStack` which re-runs `applyAttributeModifiers`, dropping the broken piece's armor/toughness/custom attributes from the entity's cache.
- `ItemStackAttributeMixin` — `@Inject(at = HEAD, cancellable = true)` on `ItemStack.applyAttributeModifiers(EquipmentSlot, BiConsumer)` (the plural overload, the one `LivingEntity.updateAttributes` iterates). Cancels when the stack is flagged `INDESTRUCTIBLE` and is at its durability cap (`getDamage() >= getMaxDamage() - 1`), so the `BiConsumer` never sees the stack's `ATTRIBUTE_MODIFIERS` entries. Broken pieces contribute no armor, no toughness, no custom modifiers — they behave as unworn.
- `ItemStackEnchantmentsMixin` — MixinExtras `@ModifyReturnValue` on `ItemStack.getEnchantments()` at `RETURN`. Returns `ItemEnchantmentsComponent.DEFAULT` when the stack is flagged `INDESTRUCTIBLE` and at the cap. **Covers the UI/tooltip path only** — tooltip rendering, the anvil/grindstone result slots, and any mod code that asks `stack.getEnchantments()`. Vanilla's enchant-effect pipeline (`EnchantmentHelper.forEachEnchantment(...)`) does **not** call `getEnchantments()`; it reads the `ENCHANTMENTS` component directly off the stack, which is why the effect-dispatch gate lives in `EnchantmentHelperForEachMixin` below. Keeping both is load-bearing: this mixin makes the broken state legible to the player, the other makes it real.
- `EnchantmentHelperForEachMixin` — `@Inject(at = HEAD, cancellable = true)` on **both** `EnchantmentHelper.forEachEnchantment(ItemStack, EnchantmentHelper$Consumer)` and `EnchantmentHelper.forEachEnchantment(ItemStack, EquipmentSlot, LivingEntity, EnchantmentHelper$ContextAwareConsumer)`. Cancels the iteration when the stack is flagged `INDESTRUCTIBLE` and at the durability cap, so no enchant effect fires for a broken piece — Frost Walker, Soul Speed, Thorns, Protection, Efficiency, Unbreaking, Mending, all of them. These two overloads are the **only** paths vanilla uses to iterate a stack's enchants for effect dispatch: the stack-only variant (reads via `stack.getOrDefault(DataComponentTypes.ENCHANTMENTS, ...)`) is called by `getDamage` / `modifyKnockback` / `getItemDamage` / `getProjectileCount` / similar math helpers; the slot-aware variant (reads via `stack.get(DataComponentTypes.ENCHANTMENTS)`) is called by `applyLocationBasedEffects` (Frost Walker's `location_changed` effect dispatches here), `onTick`, `isInvulnerableTo`, `getProtectionAmount`, `onTargetDamaged`, `onHitBlock`, and also by the entity-wide `forEachEnchantment(LivingEntity, ContextAwareConsumer)` overload which iterates the four equipment slots and funnels into it. Neither overload calls `ItemStack#getEnchantments()` — that's why `ItemStackEnchantmentsMixin` alone cannot stop effect dispatch. Stormlight + durability regen are unaffected: they read the `REGENS_DURABILITY` data component via `stack.get(ModComponents.REGENS_DURABILITY)`, which is a direct component lookup, not an enchant iteration.
- `MobEntityAccessor` — `@Accessor("targetSelector")` interface widening access to `MobEntity.targetSelector` (vanilla-`protected`). Used by `NecromancerSummon.lockTargetToAttacker` to clear every `ActiveTargetGoal` from a freshly-spawned zombie and install a single predicate-gated replacement bound to the summoner's latest-attacker UUID. Without the accessor, the zombie's vanilla `ActiveTargetGoal<PlayerEntity>` rescans every tick and retargets the summoner (the nearest player), defeating the whole mechanic. Pure getter — no behavior mixin, just a visibility widener.
- `LivingEntityEquipMixin` — `@Inject(at = HEAD)` on `LivingEntity.onEquipStack(EquipmentSlot, ItemStack, ItemStack)`. Server-only (gated on `ServerPlayerEntity`). Calls `ArmorEffects.refreshBonusesFor(player)` once — the helper walks the full `BONUSES` list and runs the `MOD_MANAGED` diff, so one call handles both APPLY (newly-equipped piece whose gate now passes) and STRIP (previously-equipped piece whose gate now fails). Replaces the previous hand-rolled enumerated strip list (Healer's Vision, Empowered Riptide, etc.) — one entrypoint, every armor-driven effect covered by the same diff path the 80-tick tick loop uses. HEAD inject is correct because `player.getEquippedStack(slot)` for armor reads `PlayerInventory.armor[i]`, which the inventory has already mutated by the time `onEquipStack` fires.
- `ServerPlayNetworkHandlerMixin` — `@Inject(at = TAIL)` on `ServerPlayNetworkHandler.onUpdateSelectedSlot(UpdateSelectedSlotC2SPacket)`. Calls `ArmorEffects.refreshBonusesFor(player)` for the hotbar-scroll / swap-key case. Same pattern as `LivingEntityEquipMixin` — one refresh call, both directions (APPLY for the newly-selected mainhand, STRIP for mainhand-driven effects whose flag just dropped off the active slot) handled via the MOD_MANAGED diff. Replaces the previous hand-rolled enumerated strip list (Emperor's Divide, God of Thunder, etc.). TAIL inject ensures the new slot index has been validated and applied before reading `player.getMainHandStack()` during refresh.
- `ServerPlayNetworkHandlerClickSlotMixin` — two TAIL `@Inject` hooks on `ServerPlayNetworkHandler`. (1) `onClickSlot(ClickSlotC2SPacket)` — the survival / standard inventory path, covers click-drag, shift-click, armor-swap-key, and any non-creative container interaction. (2) `onCreativeInventoryAction(CreativeInventoryActionC2SPacket)` — creative inventory tab drag, which does NOT route through `onClickSlot` and therefore wouldn't otherwise be caught. Without the second hook, dragging an armor piece out of the creative inventory onto the character left HUD icons for armor-driven effects lagging up to 4 s behind the actual equipment state. Both hooks call `ArmorEffects.refreshBonusesFor(player)` — same one-entrypoint pattern as the two mixins above.
- `LivingEntityRiptideLandMixin` — `@Inject(at = HEAD)` on `LivingEntity#tickRiptide(Box, Box)V`. Asgardian's Flight (Thor 5/5) post-riptide swing-input fix. Vanilla `tickRiptide` only clears `riptideTicks` on (a) hitting another `LivingEntity` mid-spin or (b) `horizontalCollision` — never on a pure vertical landing. So a straight-up/straight-down Mjolnir launch keeps `USING_RIPTIDE_FLAG` (and `EntityPose.SPIN_ATTACK`) latched for the full 20-tick riptide window after touchdown, suppressing the player's swing input — the next left-click is silently eaten until the timer elapses. The mixin sets `riptideTicks = 0` at HEAD when the receiver is a `PlayerEntity`, has `ASGARDIANS_FLIGHT`, and is currently `isOnGround()`, so the same-tick tail of `tickRiptide` (which gates the FLAG clear on `riptideTicks <= 0`) releases the spin-attack pose immediately and the next swing lands. Gated on `ASGARDIANS_FLIGHT` specifically (not on a generic ground-check) so vanilla tridents and the Poseidon dry-land riptide (`TridentItemMixin`) are unaffected — Poseidon wearers never carry that effect, so their existing horizontalCollision-only clear behavior is preserved.

Why HEAD-cancel on the anvil merge: vanilla's `updateResult` early-returns mid-method whenever the two input items aren't a valid anvil pair (chestplate + elytra aren't same-item and elytra isn't a repair material), and `@At("TAIL")` only fires at the last RETURN instruction, so a TAIL inject would never see the chestplate+elytra case. The merge inject therefore runs at HEAD with `cancellable = true` and calls `sendContentUpdates()` explicitly. The grindstone re-stamp runs at TAIL because vanilla has already computed the result stack and called `sendContentUpdates()` by then — the mixin just patches the result in place.

**Vanilla anvil / grindstone behavior on themed gear.** Outside the chestplate→elytra merge (anvil) and the merged-elytra re-stamp (grindstone), the mod does **nothing** to either screen. Themed diamond chestplates accept anvil combines exactly like a vanilla diamond chestplate would; Mjolnir is no longer anvil-locked at the mixin layer (if it accumulates weird behavior in-world, that's vanilla mace + Riptide interaction, not a mod gate); Sage's Grace grindstones exactly like a vanilla diamond sword; Shard armor grindstones exactly like vanilla diamond armor. Applies in both creative and survival. Any creative-vs-survival asymmetries the player observes (e.g. Thorns III appearing on an elytra in creative but not survival) are vanilla Mojang behavior, not mod behavior — the mod no longer intermediates.

**Enchanting table is naturally blocked** (no mixin needed): `ItemStack.isEnchantable()` returns false when the stack already has enchants, and every signature weapon ships with at least one enchant. **Smithing table (netherite upgrade) is intentionally untouched** — it uses `SmithingScreenHandler` and preserves all custom components through `SmithingTransformRecipe`.
